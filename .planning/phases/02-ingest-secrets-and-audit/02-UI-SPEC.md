# Phase 2 — UI Design Contract

> Visual and interaction contract for the spreadsheet-replacement product. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Stack reality:** SvelteKit 2.58 + Svelte 5 (runes) + Paraglide JS 2 + native CSS. No Tailwind, no shadcn, no React, no component library. Indie-budget constraint (PROJECT.md) and locked-stack constraint (RESEARCH.md §"Standard Stack") forbid adding any new top-level UI dependency in Phase 2. Tokens are CSS custom properties; components are `.svelte` files with scoped `<style>` blocks; the only global stylesheet is `src/app.css` imported once from `src/routes/+layout.svelte`.
>
> **Aesthetic stance** (PROJECT.md tone + CONTEXT.md `<specifics>`): tool-feel, terminal-leaning, dense information OK, calm > flashy, privacy-only with zero growth surfaces. Empty-state copy is **practical** (real example URLs, not marketing taglines).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla CSS, indie-budget constraint) |
| Preset | not applicable |
| Component library | none — Svelte 5 components in `src/lib/components/` |
| Icon library | inline SVG sprite (`src/lib/icons/`) — no icon-font or `lucide` dep |
| Font | system stack (no webfont — self-host parity, offline OK): `ui-sans-serif, system-ui, -apple-system, "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif` for prose; `ui-monospace, "SF Mono", Menlo, Consolas, "Liberation Mono", monospace` for example URLs (D-43) and `last4` masks |

### Why no component library

- **Indie-budget constraint** (PROJECT.md) forbids paid SaaS deps; the OSS alternatives (shadcn-svelte, bits-ui, melt-ui) are MIT but each one is a new dep with its own update cadence and a transitive Tailwind/Radix expectation that the planner cannot adopt without user sign-off.
- **Stack-lock constraint** (RESEARCH.md §"Standard Stack"): "Phase 2 introduces no new top-level deps." Adding bits-ui or melt-ui breaks that line.
- **Phase 2 is 7 page bundles** (CONTEXT.md `<code_context>` Integration Points). Each page renders forms, lists, and one modal — well within vanilla Svelte 5 reach. The cost of building 7 pages without a library is lower than the cost of vetting + locking a library + threading its theme system through D-40 cookie-driven dark mode.
- **OSS self-host parity:** no CDN-required webfonts (system stack), no analytics scripts, no Google Fonts call from the browser. The same Docker image renders identically on a self-hoster's air-gapped VPS.

## Spacing Scale

Declared values (multiples of 4, 8-point baseline). Exposed as CSS custom properties on `:root`:

```css
:root {
  --space-xs: 4px;
  --space-sm: 8px;
  --space-md: 16px;
  --space-lg: 24px;
  --space-xl: 32px;
  --space-2xl: 48px;
  --space-3xl: 64px;
}
```

| Token | Value | Usage |
|-------|-------|-------|
| `--space-xs` | 4px | Icon-text gap inside chips, gap between badge icon and label |
| `--space-sm` | 8px | Inline form-control gap (label ↔ input), gap between adjacent chips |
| `--space-md` | 16px | Default element spacing inside cards, default form-row gap, default list-row vertical padding |
| `--space-lg` | 24px | Card internal padding, gap between distinct UI clusters on a page |
| `--space-xl` | 32px | Major page-section breaks (between paste-box and timeline; between settings sub-blocks) |
| `--space-2xl` | 48px | Page-top padding on desktop, margin between unrelated regions on settings page |
| `--space-3xl` | 64px | Reserved — Phase 4 chart top/bottom breathing room (declared now to lock the scale) |

**Exceptions:**

- **Touch target minimum 44px** for icon-only buttons (theme toggle, kind-icon picker on event create, soft-delete affordance) per WCAG 2.5.5 AA. Realised as `min-height: 44px; min-width: 44px;` with internal padding adjusted; the *gap* between elements still uses the scale.
- **`<PasteBox>` input minimum height 48px** so the touch target on a 360px-wide phone (D-42) clears the 44px floor with extra forgiveness for the dominant primary action.

## Typography

Three sizes, two weights — declared as CSS custom properties on `:root`:

```css
:root {
  --font-size-body: 16px;
  --font-size-label: 14px;
  --font-size-heading: 24px;
  --font-size-display: 32px;
  --font-weight-regular: 400;
  --font-weight-semibold: 600;
  --line-height-body: 1.5;
  --line-height-heading: 1.2;
  --line-height-mono: 1.4;
}
```

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px | 400 | 1.5 | Paragraph copy, list-row body text, audit-row metadata, event notes |
| Label | 14px | 400 (form labels) / 600 (chip labels) | 1.4 | Form labels, chips, badges, table column heads, secondary metadata (timestamps, IPs) |
| Heading | 24px | 600 | 1.2 | Page heading (`<h1>` on `/games`, `/audit`, `/events`, etc.), card title |
| Display | 32px | 600 | 1.2 | Empty-state heading only (the only place big type buys clarity in a tool-feel UI) |

**Monospace exception** (D-43): example URLs in `<EmptyState>` and `last4` masks in `<KeyMaskRow>` use the monospace stack at 14px / weight 400 / line-height 1.4. This is intentional — it signals "this is a literal string you can paste" and reinforces the terminal-leaning aesthetic.

**Why not 4 sizes?** Phase 2 has no chart legends, no table footers, no tertiary labels — the three-tier list (label / body / heading) plus a single display size for empty-state poster-text covers every surface inventoried above. Phase 4 may earn a 4th size for chart axis labels; it does not earn one in Phase 2.

## Color

60/30/10 split with full dark/light tokenization (D-40, UX-01). Every color flows through CSS custom properties so the theme switch is a single class on `<html>`, no JS recolor.

```css
:root,
:root[data-theme="light"] {
  --color-bg: #fafafa;            /* dominant — page background */
  --color-surface: #ffffff;        /* secondary — cards, header, modal panels */
  --color-border: #e5e5e5;
  --color-text: #1a1a1a;
  --color-text-muted: #666666;
  --color-accent: #2563eb;         /* accent — see "reserved for" list */
  --color-accent-text: #ffffff;
  --color-destructive: #b91c1c;    /* destructive actions only */
  --color-info: #0891b2;           /* "Reddit support arrives in Phase 3" inline (D-18) — distinct from destructive */
  --color-success: #15803d;        /* save confirmation only */
}

:root[data-theme="dark"] {
  --color-bg: #0d0d0d;
  --color-surface: #171717;
  --color-border: #262626;
  --color-text: #f5f5f5;
  --color-text-muted: #a3a3a3;
  --color-accent: #60a5fa;
  --color-accent-text: #0d0d0d;
  --color-destructive: #f87171;
  --color-info: #22d3ee;
  --color-success: #4ade80;
}

@media (prefers-color-scheme: dark) {
  :root[data-theme="system"] {
    --color-bg: #0d0d0d;
    /* ... same as dark block ... */
  }
}
```

| Role | Light | Dark | Usage |
|------|-------|------|-------|
| Dominant (60%) | `#fafafa` | `#0d0d0d` | Page background, list-page surface |
| Secondary (30%) | `#ffffff` | `#171717` | Cards, header bar, modal panels, form fieldsets, audit rows |
| Accent (10%) | `#2563eb` | `#60a5fa` | Primary CTAs only (see reserved-for list below) |
| Destructive | `#b91c1c` | `#f87171` | Destructive actions only (game soft-delete confirm, event delete confirm, key remove confirm, `<InlineError>` border + icon) |
| Info | `#0891b2` | `#22d3ee` | Inline "Reddit support arrives in Phase 3" message (D-18) — visually distinct from destructive red |
| Success | `#15803d` | `#4ade80` | Reserved for save confirmation toast (no other use in P2) |

**Accent reserved for:**

1. The single primary CTA on each page: "+ New game" on `/games`, "+ New event" on `/events`, "Save key" on `/keys/steam`, "Add channel" on `/accounts/youtube`, "Save changes" on `/settings`.
2. The submit button inside the global `<PasteBox>` (D-18) — the most-used action on the game detail page.
3. The active-state underline/border on `<Nav>`.
4. Focus rings on inputs (`outline: 2px solid var(--color-accent)`).

**Accent is NOT for:**

- Tag chips (use `--color-border` background + `--color-text-muted`).
- Audit-action chips (use `--color-text-muted`; this is a log, not a dashboard).
- Soft-delete restore links (use the default text color + underline).
- "Replace" button on `<KeyMaskRow>` (use surface background + border; accent is too strong here — replacing a key is uncommon and serious).

**Privacy-first visual default** (PROJECT.md): no badges saying "share", "publish", "make public"; no green "live" indicators; no follower/wishlist count tickers in the header (those are private-by-default per PRIV-01). The audit-action chips use `--color-text-muted` rather than colored category swatches because the audit log is forensics, not engagement.

## Layout & Responsive Contract (UX-02 — 360px hard requirement)

D-42 makes 360px-wide rendering a hard CI gate (Vitest 4 browser mode). The contract:

### Breakpoints (declared, used sparingly)

```css
:root {
  --bp-mobile: 360px;   /* hard floor — UX-02 */
  --bp-tablet: 768px;
  --bp-desktop: 1024px;
}
```

- **Mobile-first.** Every page authored as if 360px is the default; `@media (min-width: 768px)` adds breathing room only.
- **No horizontal scroll at 360px.** `<html>` and `<body>` carry `overflow-x: hidden`; every component uses `min-width: 0` to defeat default flex `min-width: auto` overflow. Test asserts `document.documentElement.scrollWidth === document.documentElement.clientWidth` at 360px.
- **Every primary action reachable without zoom.** Touch targets ≥44px; primary CTAs are sticky to the bottom of the viewport on mobile when the page is scrollable (`<PasteBox>` on `/games/[gameId]`, "+ New event" on `/events`).
- **Page max-width 1024px on desktop**, centered, with `--space-md` horizontal padding on mobile and `--space-2xl` on desktop. Reading line length stays comfortable.
- **`<Nav>` collapses to a horizontally-scrollable strip on mobile** (no hamburger — only 6 destinations, all visible by overflow scroll). The active item is auto-scrolled into view on page load.
- **`<GameCard>` stacks vertically on mobile, flows to a 2-column grid at 768px and 3-column at 1024px**. Cover thumbnail caps at 96px square on mobile and 128px on desktop.
- **Tables on `/audit`:** at 360px, the audit log renders as a stacked-row card list (timestamp + action chip on row 1, IP + UA on row 2 in `--color-text-muted`); at 768px+, a real `<table>` with sortable columns. Same data, two layouts; planner picks whether to do this with CSS-grid responsive or two render branches.

### `<PasteBox>` interaction contract

The single load-bearing UI element of Phase 2.

1. **One input.** Label `m.paste_box_label()` = `Paste a URL`. Placeholder shows a rotating example: `https://youtube.com/watch?v=...` / `https://twitter.com/.../status/...`.
2. **Submit on Enter or button click.** Button label `m.ingest_cta_add()` = `Add`.
3. **Validation order (D-19 INGEST-04):**
   a. Client-side: is the string a parseable URL with a known host? If not → render `<InlineError>` immediately, no server call.
   b. If host is `reddit.com` or `redd.it` → render `<InlineInfo>` with `m.ingest_info_reddit_phase3()`. **Do not POST. Do not create any row.**
   c. Otherwise POST to `/api/items/youtube` (for YouTube hosts) or `/api/events` (for Twitter/Telegram hosts). Server runs URL parser → oEmbed fetch → INSERT, all-or-nothing in one tx (D-19).
   d. On 4xx response: render `<InlineError>` below the input with the server-supplied error key. Do not clear the input — the user may want to fix and retry.
   e. On 2xx: clear the input, scroll to the new item in the list, render a `m.toast_saved()` toast for 2s.
4. **The error and info regions never coexist.** A new submission clears both before validating the new value.
5. **Never half-write.** No retry button. No "saved partial — please try again." If the server returns 5xx, the error region shows `m.error_server_generic()` and the user re-submits; the server's `tracked_*_*` table never holds a partial row (validation happens BEFORE INSERT — D-19).

### Theme toggle interaction contract (D-40)

1. **Three states cycle:** click → `light` → click → `dark` → click → `system` → click → `light`.
2. **Icon dispatch:** sun (light), moon (dark), monitor (system). Inline SVG, 20px, in `<AppHeader>`.
3. **Click action:**
   a. Optimistically update `document.documentElement.dataset.theme` so the swap is instant (no FOUC).
   b. POST `/api/me/theme` with the new value. Server writes the `__theme` cookie (`Path=/; SameSite=Lax; Max-Age=31536000`, no `HttpOnly` per RESEARCH.md so client-side toggle can read it back) AND `user.theme_preference`.
   c. Audit log writes `theme.changed` with `{from, to}` metadata.
   d. On server failure: revert the `data-theme` attribute and render an inline error in the toggle's tooltip area.
4. **SSR-flash mitigation** (RESEARCH.md `transformPageChunk`): `src/hooks.server.ts` reads the `__theme` cookie before any handler runs and rewrites a `%theme%` placeholder inside `<html data-theme="%theme%">` in `app.html` via the SvelteKit `transformPageChunk` hook. The browser never sees a flash of the wrong theme.
5. **`prefers-color-scheme` honored only when `data-theme="system"`.** A `@media (prefers-color-scheme: dark) :root[data-theme="system"]` block flips the tokens.

### Form interaction contracts (every form on every page)

- **Required fields marked with `*` after the label**, not with red asterisks (red is reserved for destructive).
- **Validation runs on submit.** No live validation as the user types (PROJECT.md tone — no flashy real-time feedback). Errors render below each field with `<InlineError>`.
- **Submit button shows a spinner state and is disabled while in flight.** No double-submit possible.
- **Success state navigates back to the list view and renders a toast.** No "saved!" inline confirmation that hangs around — the toast disappears in 2s.
- **All form labels and field hints flow through Paraglide.** No hard-coded English strings in `.svelte` files (P1 invariant).
- **Cancel button is text-link styled, not a secondary button.** This deemphasizes "back out" relative to the primary save action and keeps the form visually focused.

### Loading & empty contract

- **Every list page has three states:** loading skeleton (3 placeholder rows with shimmering border-color animation, no spinner), empty state (`<EmptyState>` with example URL), or populated.
- **No spinners on initial page load** (SvelteKit SSR delivers data with the page); spinners only appear on user-initiated actions (form submit, paste-box submit).
- **Skeleton rows respect dark mode tokens** — they're not flat gray, they use `--color-border` and a subtle `--color-surface → --color-bg` gradient animation.

## Accessibility Floor

Not a separate dimension in the gsd-ui-checker rubric, but called out because it affects every component above:

- **All interactive elements use semantic HTML.** `<button>` for actions, `<a href>` for navigation, `<dialog>` for confirms, `<select>` for dropdowns. No `<div onclick>`.
- **Focus-visible outlines** are 2px solid `--color-accent` with 2px offset on every focusable element. Never removed.
- **Color contrast** light theme: `--color-text` (#1a1a1a) on `--color-bg` (#fafafa) = 16.1:1 (AAA). Dark theme: `--color-text` (#f5f5f5) on `--color-bg` (#0d0d0d) = 17.3:1 (AAA). Accent on accent-text light: `#2563eb` on `#ffffff` = 5.17:1 (AA). Accent on accent-text dark: `#60a5fa` on `#0d0d0d` = 7.93:1 (AAA). All chip / badge combinations stay ≥4.5:1.
- **Form labels are always programmatically associated** via `for` / `id` (or wrapped `<label><input/></label>`). Required fields marked with both visual `*` and `aria-required="true"`.
- **Audit log rows use `<time datetime>` for timestamps** so screen readers announce them correctly.
- **`<ConfirmDialog>` traps focus** via the native `<dialog>` element's modal behaviour (`showModal()`); cancel button is the default focus target.

## Flags (non-blocking observations for the planner)

These do not block the contract but should surface in the plan-phase agent's notes:

- **FLAG: D-43 says example URLs are "inert" (clicking does nothing).** The contract above respects this: example URLs are styled monospace inside `<EmptyState>` but they are `<code>` elements, not anchors. **However**, the principle of least surprise says hover-cursor on a URL-shaped string typically suggests clickability. Recommendation for executor: render with `cursor: text` (not `cursor: pointer`) and add a quiet tooltip `Example only — copy and paste into the box above` so the inertness is honest, not hidden.

- **FLAG: D-40 cookie name is `__theme`.** Two leading underscores is the [Cookie Prefixes spec](https://datatracker.ietf.org/doc/html/draft-ietf-httpbis-rfc6265bis) reserved namespace; `__Host-` and `__Secure-` are the two well-known prefixes. A bare `__theme` does not collide with either, but a future security-tightening pass might add `__Host-theme` (which forces `Path=/`, `Secure`, and no `Domain`). Phase 6 polish phase is the right time to revisit; not a Phase 2 blocker.

- **FLAG: The 60/30/10 color split in this spec uses a near-white surface and a true-white card on light theme.** The 60/30 contrast is intentionally subtle (`#fafafa` vs `#ffffff`) because the dense / terminal-leaning aesthetic dislikes loud panel boundaries. If checker scoring penalizes the 1% lightness gap, the alternative is `#f5f5f5` for dominant and `#ffffff` for surface — that's a 5% gap and reads as a more "designed" UI but moves further from the tool-feel stance. Planner picks; both are tokenized so the swap is one-line.

- **FLAG: No icon library means every icon ships as inline SVG.** The inventory above references icons for `<KindIcon>` (7 event kinds), theme toggle (3 icons), and a small fixed set of action icons (delete, restore, edit, copy, ext-link, plus a chevron for nav active state). Roughly 15 SVGs. The planner should consider whether to centralize these in `src/lib/icons/icons.svg` as a sprite (one HTTP request, `<use>` references) or as individual `.svelte` icon components. Sprite is cheaper at runtime; component-per-icon is easier to tree-shake. No new dep either way.

- **FLAG: Phase 2 has no tagging-input UI** (the user types an existing tag they already have). Tag chips on game cards display only — they are sourced from Steam appdetails (`genres`, `categories`, `steam_tags` from D-26 / Steam appdetails contract). User-private "campaign" tags are explicitly out of scope (CONTEXT.md `<deferred>`). The contract confirms: tag chips are read-only display, no chip-input control.
