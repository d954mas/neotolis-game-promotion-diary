## Decisions Made

(Recorded in detail in frontmatter `key-decisions`.)

- **9-key starter dictionary** — `app_title` and `dashboard_title` both resolve to `Promotion diary` in MVP but are kept as separate keys because the page `<title>` and `<h1>` are different surfaces that can diverge per future product decision (e.g. SEO would want a longer `<title>`). Not a DRY violation.
- **Parameterized welcome message** — `dashboard_welcome_intro` carries `{name}`, demonstrating Paraglide's typed-parameter support and setting the convention for all future parameterized strings. The argument flows through Plan 06's `UserDto.name`, preserving P3 DTO discipline at the rendering layer.
- **Runtime resolution test wrapped in try/catch with JSON-fallback** — the compiled `src/lib/paraglide/messages.js` is gitignored; the test passes whether or not `pnpm build` has run. CI exercises the live import path; local fresh-clone tests pass via the JSON contract assertion.
- **Explicit keyset toEqual instead of toMatchSnapshot for VALIDATION 19** — more durable across renames; PR diffs are easier to read; no snapshot file to keep in sync.
- **Integration test stays at the static-contract layer** — full end-to-end "render the page through Hono and grep English HTML" lives in Plan 10's Docker smoke test (which boots the actual built image). Fast deterministic integration test + slow comprehensive smoke test = the right split.
- **No locale-add demo in tests** — actually placing a `messages/ru.json` fixture and running Paraglide compile would require booting Vite or invoking the Paraglide CLI inside the test. The keyset invariant is the correct unit-level proxy: it asserts the contract that any future ru.json must honor, without taking on a runtime dependency on the compile step.
