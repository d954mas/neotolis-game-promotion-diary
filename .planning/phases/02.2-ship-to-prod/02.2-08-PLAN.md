# Backups (opt-in, self-host friendly)

The application has zero awareness of backups. `scripts/backup.sh` is a
standalone shell helper that pg_dumps the postgres container, gzips, and
uploads to any S3-compatible remote via rclone. Self-host operators who
don't want backups simply skip this setup; the app runs identically with

## Why opt-in
- D-25a: backup is server-side cron, not bundled in `docker-compose.selfhost.yml`.
- App + database run identically with or without backups (no env-var coupling).
- Operators choose their preferred S3 remote: Cloudflare R2 (recommended),
  Backblaze B2, Wasabi, AWS S3, MinIO (local), or no remote (skip section).

```

## §6 — Restore procedure

```bash
# Pull the latest dump
rclone copy <remote>:<bucket>/<date>.sql.gz /tmp/

# Restore (DESTRUCTIVE — replaces current DB content)
docker exec -i diary_postgres pg_restore -U postgres -d neotolis -c < /tmp/<date>.sql.gz
```

For partial restoration (single table), use pg_restore -t flag.

## §7 — Failure modes
- rclone misconfigured → script exits non-zero; `tail /var/log/diary-backup.log`.
- Disk full → /tmp dump fails; clean /tmp.
- Concurrent invocation → flock prevents (script exits 0 silently).
- Postgres container down → `docker exec` fails; check `docker compose ps`.
```
