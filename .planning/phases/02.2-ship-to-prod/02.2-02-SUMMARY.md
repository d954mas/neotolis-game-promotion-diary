## Decisions Made

- **Soft-delete exclusion is per-kind.** `games` and `data_sources` counts filter `isNull(<table>.deletedAt)` because the limits are footprint caps — soft-deleted rows occupy no live registry slot, so blocking re-creation would be perverse. `events_per_day` does NOT filter `deletedAt`: it's a rate cap, and an abusive user soft-deleting events to unlock more 24h slots would defeat the cap. Locked in RESEARCH §1; confirmed in CONTEXT D-11 (active rows; soft-deleted excluded specifically for the footprint kinds).
- **`createEvent` assertQuota lands BEFORE the kind/title/occurredAt validators.** Validate-first INSERT discipline (D-19) lets the 429 fire before the synchronous URL parse runs. Cheap rejection saves CPU on a quota-exceeded user; no quota-related state change happens between the throw and the would-be INSERT.
- **PUTOFF Phase 3 comment marks the auto-import code-path follow-up.** When the polling adapter lands in Phase 3, hitting `events_per_day` should THROTTLE (defer the polling job for retry) — NOT throw — because the source-of-truth content was already published externally and we want to ingest it eventually. Phase 2.2 only has the user-facing path; the throw is correct here.
