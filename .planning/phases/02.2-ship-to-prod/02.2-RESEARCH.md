### 1. Per-user quota numbers (D-11)

**Locked values** (planner adopts; env vars use these as zod defaults):

| Env var | Value | Rationale |
|---------|-------|-----------|
| `LIMIT_GAMES_PER_USER` | **50** | A heavy indie dev tracks 5–10 games simultaneously; 50 gives 5–10× headroom. Above 50 is almost certainly abuse / not a real indie dev. |
| `LIMIT_SOURCES_PER_USER` | **50** | A heavy indie dev with 10 games × 5 sources/game = 50 sources. Above 50 is almost certainly abuse. (Originally proposed 20 in CONTEXT.md additional-context, but 50 matches the games-per-user logic — CONTEXT.md treated 20 as "20 typical, cap at 50?" — locking the 50 ceiling here.) |
| `LIMIT_EVENTS_PER_DAY` | **500** | Manual paste during a launch week could hit 100/day; auto-import in Phase 3 may bump it. 500/day rolling 24h is a safety cap, not a throttle. CONTEXT.md additional-context proposed "500 default" — adopting that. |

**Implementation pattern (cite this verbatim in plans):**

```typescript
// src/lib/server/services/quota.ts (NEW FILE)
import { and, eq, isNull, gte, count } from "drizzle-orm";
import { db } from "../db/client.js";
import { games } from "../db/schema/games.js";
import { dataSources } from "../db/schema/data-sources.js";
import { events } from "../db/schema/events.js";
import { writeAudit } from "../audit.js";
import { env } from "../config/env.js";
import { AppError } from "./errors.js";

type QuotaKind = "games" | "data_sources" | "events_per_day";

const LIMITS: Record<QuotaKind, number> = {
  games: env.LIMIT_GAMES_PER_USER,
  data_sources: env.LIMIT_SOURCES_PER_USER,
  events_per_day: env.LIMIT_EVENTS_PER_DAY,
};

async function currentCount(userId: string, kind: QuotaKind): Promise<number> {
  if (kind === "games") {
    const [r] = await db.select({ c: count() }).from(games)
      .where(and(eq(games.userId, userId), isNull(games.deletedAt)));
    return Number(r?.c ?? 0);
  }
  if (kind === "data_sources") {
    const [r] = await db.select({ c: count() }).from(dataSources)
      .where(and(eq(dataSources.userId, userId), isNull(dataSources.deletedAt)));
    return Number(r?.c ?? 0);
  }
  // events_per_day — rolling 24h count, NOT calendar-day-server-time.
  // Avoids midnight cliff: a user posting 499 at 23:59 and 1 at 00:01 should not pass.
  const since = new Date(Date.now() - 24 * 60 * 60 * 1000);
  const [r] = await db.select({ c: count() }).from(events)
    .where(and(eq(events.userId, userId), gte(events.createdAt, since)));
  return Number(r?.c ?? 0);
}

/** Throws AppError 429 with structured body when quota exceeded. */
export async function assertQuota(
  userId: string,
  kind: QuotaKind,
  ipAddress: string,
): Promise<void> {
  const limit = LIMITS[kind];
  const current = await currentCount(userId, kind);
  if (current >= limit) {
    await writeAudit({
      userId, action: "quota.limit_hit", ipAddress,
      metadata: { kind, limit, current },
    });
    throw new AppError(
      `quota_exceeded: ${kind} ${current}/${limit}`,
      "quota_exceeded",
      429,
      { kind, limit, current },
    );
  }
}
```

**Call-site pattern:**
- `createGame` calls `await assertQuota(userId, "games", ipAddress)` BEFORE `db.insert(games)`.
- `createSource` calls `await assertQuota(userId, "data_sources", ipAddress)` BEFORE `db.insert(dataSources)`.
- `createEvent` (and the auto-import code-path in Phase 3) calls `await assertQuota(userId, "events_per_day", ipAddress)`. For the auto-import path, hitting the quota should throttle the polling adapter, NOT throw — that path is Phase 3, document via PUTOFFS comment.
- HTTP boundary already maps `AppError(429)` automatically via `mapErr` in `routes/_shared.ts` (verified line 50-72) — no new mapping needed. Body: `{error: 'quota_exceeded', metadata: {kind, limit, current}}`.

**Reset semantics:** rolling 24h via `WHERE created_at > now() - interval '24 hours'`. Locked.

**Audit:** `quota.limit_hit` with `metadata: {kind, limit, current}` written every time a guard fires. Operator (author) inspects `audit_log` to discover real users approaching limits and tune up via env var without redeploying code.

**Architecture:** ship two files in repo, plus a tiny refresh helper.

```
nginx/

```

**`nginx/nginx.conf.template`** (locked shape):

```nginx
# Cloudflare's "real client IP" — must precede the first set_real_ip_from
real_ip_header CF-Connecting-IP;
real_ip_recursive on;

# Cloudflare published IP ranges (sourced from cf-ips.conf — refreshed
# quarterly per docs/deploy/install.md §"Refresh CF IPs").
include /etc/nginx/cf-ips.conf;

# Per-IP rate limit on auth endpoints (D-28: 10 req/sec per IP).
# 10 req/sec, burst 20, no-delay (immediate 503 on burst overrun).
limit_req_zone $binary_remote_addr zone=auth:10m rate=10r/s;

# Hide nginx version from response headers
server_tokens off;

server {
    listen 80;
    server_name ${DOMAIN};

    # Resolver for Docker network DNS (app:3000)
    resolver 127.0.0.11 valid=10s ipv6=off;

    # gzip on for HTML / JSON / CSS / JS responses (D-Discretion item)
    gzip on;
    gzip_vary on;
    gzip_proxied any;
    gzip_comp_level 6;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/rss+xml;
    gzip_min_length 1024;

    # Security headers (CF also sets HSTS at the edge; defense-in-depth here too)
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;

    # Rate-limited auth endpoints (Better Auth + OAuth callback)
    location /api/auth/ {
        limit_req zone=auth burst=20 nodelay;
        proxy_pass http://app:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        # Forward CF-Connecting-IP through (real_ip_header above already
        # rewrote $remote_addr; the application's proxy-trust middleware
        # also reads CF-Connecting-IP — both are correct, double-checked).
        proxy_set_header CF-Connecting-IP $http_cf_connecting_ip;
        proxy_read_timeout 30s;
        proxy_send_timeout 30s;
    }

    # Everything else
    location / {
        proxy_pass http://app:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header CF-Connecting-IP $http_cf_connecting_ip;
        proxy_cache_bypass $http_upgrade;
        # 60s read timeout matches the export-account streaming response
        # for heavy users (~10MB envelope; comfortable margin).
        proxy_read_timeout 60s;
        proxy_send_timeout 60s;
        proxy_connect_timeout 10s;
    }
}
```

### 3. VPS hardening recipe (D-28)

**Locked sequence (Ubuntu 24.04 LTS aeza default):**

```bash
# 1. apt update + automatic security updates
apt update && apt full-upgrade -y
apt install -y unattended-upgrades fail2ban ufw
dpkg-reconfigure --priority=low unattended-upgrades  # interactive: enable

# 2. UFW firewall (default deny inbound; allow ssh rate-limited + 80 + 443)
ufw default deny incoming
ufw default allow outgoing
ufw limit 22/tcp                  # rate-limit ssh (UFW's built-in: 6/min/source)
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable

# 3. Non-root deploy user
useradd -m -s /bin/bash deploy
usermod -aG docker deploy         # docker group is created by docker-ce install
mkdir -p /home/deploy/.ssh
# (manually paste author's public key into /home/deploy/.ssh/authorized_keys)
chmod 700 /home/deploy/.ssh
chmod 600 /home/deploy/.ssh/authorized_keys
chown -R deploy:deploy /home/deploy/.ssh

# 4. SSH hardening (after deploy user is set up + tested)
sed -i 's/^#\?PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config
sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sed -i 's/^#\?KbdInteractiveAuthentication.*/KbdInteractiveAuthentication no/' /etc/ssh/sshd_config
systemctl restart ssh

# 5. fail2ban — sshd jail enabled by default on Ubuntu 24.04;
#    confirm and (optionally) tune findtime/bantime
cat > /etc/fail2ban/jail.d/sshd.local <<EOF
[sshd]
enabled = true
maxretry = 5
findtime = 10m
bantime = 1h
EOF
systemctl restart fail2ban

# 6. Docker daemon log rotation
mkdir -p /etc/docker
cat > /etc/docker/daemon.json <<EOF
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
EOF
systemctl restart docker

# 7. Auto-recovery on reboot (D-22a — verify defaults are correct)
systemctl enable docker     # already enabled by default on 24.04, but confirm
systemctl enable cron       # already enabled by default on 24.04, but confirm
systemctl is-enabled docker # expect: enabled
systemctl is-enabled cron   # expect: enabled
```

**`scripts/deploy.sh`:**

```bash
#!/usr/bin/env bash
set -euo pipefail

DEPLOY_USER="${DEPLOY_USER:-deploy}"
DEPLOY_HOST="${DEPLOY_HOST:?DEPLOY_HOST required (e.g. diary.example.app)}"
APP_DIR="${APP_DIR:-/opt/diary}"

echo "==> Triggering deploy on ${DEPLOY_USER}@${DEPLOY_HOST}"

# The remote SSH key is `command="..."` restricted (see VPS hardening §3).
# So this is just opening an SSH session — the restricted command runs
# automatically. We pipe through ssh and capture the output.
ssh -o BatchMode=yes -o StrictHostKeyChecking=accept-new \
    "${DEPLOY_USER}@${DEPLOY_HOST}" 2>&1 | tee deploy.log

echo "==> Deploy triggered. Verifying health..."
sleep 15
if curl -fsS "https://${DEPLOY_HOST}/healthz" >/dev/null; then
```

### Soft-cascade for account deletion (clones `softDeleteGame` from `services/games.ts` line 269-310)

```typescript
// src/lib/server/services/account.ts (NEW FILE)
//
// Account-level soft-delete cascades to ALL user-owned tables in ONE
// transaction with a single captured Date (D-23 pattern from games.ts).
// On restoreAccount, only children whose deletedAt === user.deletedAt
// are reversed — children soft-deleted EARLIER stay deleted (the
// marker-timestamp design).
//
// Pattern 1: every function takes userId first; every Drizzle query
// filters eq(<table>.userId, userId). Tenant-scope ESLint rule fires
// on the absence of this filter.
//
// Cascade tables (verified 2026-05-01 via src/lib/server/db/schema/index.ts):
//   - user (parent — Better Auth user table; needs deleted_at column added)
//   - games
//   - game_steam_listings
//   - data_sources
//   - events
//   - api_keys_steam
//   - audit_log: NOT cascaded — INSERT-only invariant (AGENTS.md §4)
//   - session: hard-deleted, not soft-deleted (signOutAllDevices)
//   - account (Better Auth OAuth tokens): hard-deleted; nothing to restore
//   - event_games: schema CASCADE on game_id (no deleted_at column;
//     events stay attached to games via the junction; soft-delete of
//     parent leaves junction rows alone)

import { and, eq, isNull } from "drizzle-orm";
import { db } from "../db/client.js";
import { user, session, account } from "../db/schema/auth.js";
import { games } from "../db/schema/games.js";
import { gameSteamListings } from "../db/schema/game-steam-listings.js";
import { dataSources } from "../db/schema/data-sources.js";
import { events } from "../db/schema/events.js";
import { apiKeysSteam } from "../db/schema/api-keys-steam.js";
import { writeAudit } from "../audit.js";
import { env } from "../config/env.js";
import { NotFoundError, AppError } from "./errors.js";

export async function softDeleteAccount(
  userId: string,
  ipAddress: string,
): Promise<void> {
  const deletedAt = new Date();
  await db.transaction(async (tx) => {
    // Parent
    const result = await tx.update(user)
      .set({ deletedAt })
      .where(and(eq(user.id, userId), isNull(user.deletedAt)))
      .returning({ id: user.id });
    if (result.length === 0) throw new NotFoundError();

    // Children (all 5 user-owned tables)
    await tx.update(games).set({ deletedAt })
      .where(and(eq(games.userId, userId), isNull(games.deletedAt)));
    await tx.update(gameSteamListings).set({ deletedAt })
      .where(and(eq(gameSteamListings.userId, userId), isNull(gameSteamListings.deletedAt)));
    await tx.update(dataSources).set({ deletedAt })
      .where(and(eq(dataSources.userId, userId), isNull(dataSources.deletedAt)));
    await tx.update(events).set({ deletedAt })
      .where(and(eq(events.userId, userId), isNull(events.deletedAt)));
    await tx.update(apiKeysSteam).set({ deletedAt })
      .where(and(eq(apiKeysSteam.userId, userId), isNull(apiKeysSteam.deletedAt)));

    // Hard-delete sessions (forces logout on next request).
    // account (OAuth tokens) — leave alone; on restore, user can re-link
    // via OAuth login. Hard-deleting account would break Better Auth's
    // re-link flow on restore.
    await tx.delete(session).where(eq(session.userId, userId));
  });

  await writeAudit({
    userId, action: "account.deleted", ipAddress,
    metadata: { retentionDays: env.RETENTION_DAYS },
  });
}

export async function restoreAccount(
  userId: string,
  ipAddress: string,
): Promise<void> {
  await db.transaction(async (tx) => {
    const [parent] = await tx.select({ deletedAt: user.deletedAt })
      .from(user).where(eq(user.id, userId)).limit(1);
    if (!parent || parent.deletedAt === null) throw new NotFoundError();
    const markerTs = parent.deletedAt;

    // Within RETENTION_DAYS?
    const ageDays = (Date.now() - markerTs.getTime()) / (1000 * 60 * 60 * 24);
    if (ageDays > env.RETENTION_DAYS) {
      throw new AppError(
        "account_purge_window_expired",
        "account_purge_window_expired",
        410,  // Gone
      );
    }

    // Reverse parent
    await tx.update(user).set({ deletedAt: null })
      .where(eq(user.id, userId));

    // Reverse only children whose deletedAt matches the marker
    await tx.update(games).set({ deletedAt: null })
      .where(and(eq(games.userId, userId), eq(games.deletedAt, markerTs)));
    await tx.update(gameSteamListings).set({ deletedAt: null })
      .where(and(eq(gameSteamListings.userId, userId), eq(gameSteamListings.deletedAt, markerTs)));
    await tx.update(dataSources).set({ deletedAt: null })
      .where(and(eq(dataSources.userId, userId), eq(dataSources.deletedAt, markerTs)));
    await tx.update(events).set({ deletedAt: null })
      .where(and(eq(events.userId, userId), eq(events.deletedAt, markerTs)));
    await tx.update(apiKeysSteam).set({ deletedAt: null })
      .where(and(eq(apiKeysSteam.userId, userId), eq(apiKeysSteam.deletedAt, markerTs)));
  });

  await writeAudit({ userId, action: "account.restored", ipAddress });
}

export async function exportAccountJson(userId: string): Promise<{
  exported_at: string;

  api_keys_steam: import("../dto.js").ApiKeySteamDto[];

}> {

  // ... implementation reads each table scoped by userId, maps through
  // its toEntityDto, returns the envelope.
```

## Common Pitfalls

### Pitfall 1: GHCR image visibility defaults to private

**What goes wrong:** First push to `ghcr.io/d954mas/neotolis-diary` creates a private image. Self-host operators get `unauthorized` on `docker pull` even though the Git repo is public.

**Why it happens:** GHCR follows GitHub's per-package visibility model, not the repo's visibility. New packages start private regardless of source repo visibility.

**How to avoid:** Document a one-time UI step in `docs/deploy/install.md` after the first CI build:
1. Go to `https://github.com/d954mas?tab=packages` (or org URL).
2. Click `neotolis-diary` package → "Package settings" → "Change visibility" → "Public".

**Warning sign:** A self-host user reports `docker pull ghcr.io/d954mas/neotolis-diary:latest` returns 401.

### Pitfall 2: CF orange-cloud caches `/api/auth/callback/google`

**What goes wrong:** Cloudflare's edge cache may serve a stale OAuth callback response, breaking subsequent logins.

**Why it happens:** CF caches based on URL + cookies; OAuth callbacks have query-string state but no `Cache-Control: no-store` from Better Auth.

**How to avoid:** Add a CF Page Rule for `<domain>/api/*` with "Cache Level = Bypass". Document in install.md.

**Warning sign:** Sporadic OAuth failures ("invalid state token") for new users.

### Pitfall 3: `OAUTH_REDIRECT_URI` mismatch with Google Cloud Console

**What goes wrong:** Production OAuth login fails with "redirect_uri_mismatch" error from Google.

**Why it happens:** The redirect URI in Google Cloud Console must EXACTLY match what Better Auth sends; trailing slash, http vs https, port number all matter.

**How to avoid:** The redirect URI is `https://<DOMAIN>/api/auth/callback/google` (no trailing slash). Document in install.md as a verbatim copy-paste step. The `OAUTH_PROVIDER_ID=google` default in env.ts (verified line 34) means Better Auth's path is `/api/auth/callback/google`.

**Warning sign:** First login attempt redirects back with "Error: redirect_uri_mismatch".

### Pitfall 4: Better Auth session cleanup on account-delete

**What goes wrong:** User's existing browser tabs continue to show authenticated UI for ~minutes after account-delete because Better Auth caches the session in-memory.

**Why it happens:** Better Auth's session cache is per-process; `signOutAllDevices` deletes DB rows but doesn't invalidate the in-memory cache on other workers.

**How to avoid:** The DB row deletion is the load-bearing barrier — Better Auth's session validation re-reads the DB on every request (verified Phase 1 STATE.md: "AUTH-02 specifically requires server-side session invalidation — we DELETE the row instead of marking it expired so the same cookie value can never resurrect a session"). The next request from a stale tab will 401. Document the ~1-second propagation in install.md.

**Warning sign:** Test asserts that after `DELETE /api/me/account`, the next request with the same cookie gets 401.

### Pitfall 5: Drizzle migration ordering with concurrent branches

**What goes wrong:** Two PRs both add migration `0008_*.sql`; one merges first; the other has to renumber. If un-renumbered, drizzle-kit reports schema drift on boot.

**Why it happens:** Forward-only migrations have a strict numeric ordering captured in `drizzle/meta/_journal.json`.

**How to avoid:** Rebase the PR before merge; rename the migration file; regenerate `_journal.json`. Document in `docs/deploy/install.md` rollback section ("if migration fails on boot, this is what to do").

**Warning sign:** `pnpm db:check` fails in CI on master after merge.

### Pitfall 6: R2 Object Lock cannot be enabled without versioning... wait, actually it can

**What goes wrong:** Operators following old AWS S3 docs assume R2 needs versioning + Object Lock at bucket-creation time.

**Why it happens:** R2 calls the feature **bucket locks** (not Object Lock). Per CF docs (verified 2026-05-01: `https://developers.cloudflare.com/r2/buckets/bucket-locks/`), bucket locks **CAN be enabled on existing buckets** via Wrangler CLI or dashboard. Versioning is NOT required.

**How to avoid:** Document in `docs/self-host/backups.md` the Wrangler CLI command:
```bash
npx wrangler r2 bucket lock add diary-backups --prefix '' --age-seconds 2592000
# 2592000 seconds = 30 days; matches D-25 retention
```

**Warning sign:** Operator googles "R2 Object Lock" and finds AWS docs that don't apply.

### Pitfall 7: `restart: unless-stopped` vs `always`

**What goes wrong:** A deliberate `docker compose stop` (e.g. for maintenance) is overridden on host reboot, restarting containers operator wanted off.

**Why it happens:** `restart: always` ignores operator stop intent on reboot.

**How to avoid:** Use `restart: unless-stopped` (D-22a explicit choice). Locked.

**Warning sign:** Operator stops the app for maintenance and finds it back up after a host reboot.

### Pitfall 8: Multiple cron entries (operator copy-paste mistake)

**What goes wrong:** Operator follows install.md and self-host/backups.md both, ends up with two crontab entries for backup; both fire at 3 AM; flock prevents the duplicate but log noise + operator confusion.

**Why it happens:** Documentation duplication; operator runs `crontab -e` twice.

**How to avoid:** Both runbooks reference the same canonical line `0 3 * * * /usr/local/bin/diary-backup >> /var/log/diary-backup.log 2>&1`. The `flock -n /var/lock/diary-backup.lock` at the top of `backup.sh` ensures the second invocation exits 0 silently.

**Warning sign:** `journalctl | grep diary-backup` shows two firings within seconds.

### Pitfall 9: Self-host parity grep regression

**What goes wrong:** A future PR adds `if (env.APP_MODE === 'saas') { ... }` for some Phase 02.2 feature; CI smoke gate doesn't catch it because the smoke uses `selfhost.yml`.

**Why it happens:** Subtle code that branches on APP_MODE wins past the smoke gate when the SaaS-specific code path is empty in selfhost.

**How to avoid:** Extend the SaaS-leak grep step in CI (D-30) to fail on `APP_MODE === 'saas'` literal in src/. Plan 02.2-07 must add this grep step.

**Warning sign:** Self-host users report a feature missing that "works for the author".

### Pitfall 10: Privacy Policy claims about encryption don't match reality

**What goes wrong:** Privacy Policy says "OAuth tokens are encrypted at rest" but Better Auth's `account` table stores them as plaintext (verified `src/lib/server/db/schema/auth.ts` lines 56-58: `accessToken: text("access_token")`, `refreshToken: text("refresh_token")`, `idToken: text("id_token")` — all `text`, not `bytea`).

**Why it happens:** Phase 1 didn't envelope-encrypt OAuth tokens; KEYS-* tables (Phase 2) do. Privacy Policy boilerplate may overclaim.

**How to avoid:** Privacy Policy MUST accurately describe what's encrypted: "Per-user API keys for YouTube/Reddit/Steam are envelope-encrypted at rest. OAuth refresh tokens are stored in the database (TLS-encrypted in transit; not envelope-encrypted at rest in this version)." OR change the schema in Phase 02.2 to envelope-encrypt OAuth tokens too — but that's a larger surface and Phase 6 territory. Recommend: accurate disclosure now; encryption upgrade in a later phase.

**Warning sign:** A user reads the Privacy Policy and asks the author "what does envelope encryption mean for my Google refresh token?" and the answer is "actually that one isn't encrypted at rest yet".

### Pitfall 11: Build-time env placeholders leak into Dockerfile

**What goes wrong:** Looking at the existing Dockerfile (lines 33-40), build-stage env placeholders include a base64-encoded KEK. If an attacker pulls the image and runs `docker history --no-trunc`, they could see the placeholder values. Not a real KEK, but ugly.

**Why it happens:** SvelteKit build needs env vars to be present for module init; placeholders are necessary.

**How to avoid:** Already mitigated — the runtime stage starts FROM `node:22-alpine` fresh, NOT from the build stage; placeholder env vars don't propagate (verified Dockerfile line 46: `FROM node:22-alpine AS runtime`). No action needed; document in plan as "verified via existing Dockerfile review".

### Pitfall 12: Better Auth's `user` schema lacks `deleted_at`

**What goes wrong:** `softDeleteAccount` fails at compile time because `user.deleted_at` doesn't exist.

**Why it happens:** Verified `src/lib/server/db/schema/auth.ts` lines 15-28 — the `user` table has only `id`, `email`, `emailVerified`, `name`, `image`, `themePreference`, `createdAt`, `updatedAt`. No `deletedAt`.

**How to avoid:** Migration `0008_*.sql` adds the column:
```sql
ALTER TABLE "user" ADD COLUMN "deleted_at" timestamp with time zone;
```
Plus update `src/lib/server/db/schema/auth.ts` to add `deletedAt: timestamp("deleted_at", { withTimezone: true })`.

**Verify with `pnpm db:check`:** drizzle-kit detects the schema-vs-DB drift and points at the change.

**Warning sign:** TypeScript error: `Property 'deletedAt' does not exist on type 'PgTableWithColumns<...>'`.

### Example 2: `scripts/backup.sh` (locked shape, ~30 lines)

```bash
#!/usr/bin/env bash
# diary-backup — daily Postgres dump → R2 push (D-25, D-25b).
#
# Idempotent: flock prevents concurrent runs. Same-day filename means
# multiple runs the same day overwrite the same R2 key; bucket-lock
# versioning preserves the prior version.
#
# Failures DO NOT page the operator (D-25b — UptimeRobot pings /healthz,
# not backup status). Operator inspects R2 manually monthly.
#
# rclone reads its own config from ~/.config/rclone/rclone.conf; the
# application has zero awareness of backup credentials (D-25a).

set -euo pipefail

LOCK="/var/lock/diary-backup.lock"
DATE="$(date -u +%Y-%m-%d)"
DUMP_FILE="/tmp/diary-backup-${DATE}.sql.gz"
RCLONE_REMOTE="${RCLONE_REMOTE:-r2}"
RCLONE_BUCKET="${RCLONE_BUCKET:-diary-backups}"
PG_CONTAINER="${PG_CONTAINER:-diary_postgres}"

# flock: -n = non-blocking; exit 0 silently if held (no error noise).
exec 200>"$LOCK"
flock -n 200 || { echo "[$(date -u)] backup already running, skipping"; exit 0; }

echo "[$(date -u)] starting backup ${DATE}"

# pg_dump runs INSIDE the postgres container (no postgresql-client
# install on the host required). -Fc = custom format; restored via
# pg_restore. gzip on the host side.
docker exec "$PG_CONTAINER" pg_dump -Fc -U postgres neotolis | gzip > "$DUMP_FILE"
SIZE=$(stat -c%s "$DUMP_FILE")
echo "[$(date -u)] dump size: ${SIZE} bytes"

# Upload via rclone rcat (stream upload, no local file copy needed).
# Actually we already have the file; rclone copyto preserves the name.
rclone copyto "$DUMP_FILE" "${RCLONE_REMOTE}:${RCLONE_BUCKET}/${DATE}.sql.gz"

rm -f "$DUMP_FILE"
echo "[$(date -u)] backup ${DATE} OK (${SIZE} bytes)"
```

**Crontab entry (in install.md):**
```
0 3 * * * /usr/local/bin/diary-backup >> /var/log/diary-backup.log 2>&1
```

| Old Approach | Current Approach | When Changed | Impact |
|---|---|---|---|
| Build image on-server (`game_idea_founder` deploy.sh runs `docker compose ... up -d --build` over ssh) | Build image in CI → push to GHCR → server pulls | Phase 02.2 | Faster deploys (~30s vs ~3min); reproducible builds; rollback by re-pulling earlier sha |
