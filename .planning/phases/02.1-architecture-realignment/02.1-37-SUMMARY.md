# Phase 02.1 Plan 37: Merged-state validator + /feed kindList filter Summary

**PATCH /api/events/:id now validates the kind=youtube_video → URL invariant against the MERGED state at the service layer; /feed?kind=foo is silently dropped before reaching Drizzle's inArray clause; both round-6 gaps (UAT-NOTES.md §5.11 P1 + §5.13 P2) closed.**

## Accomplishments

- **Closed UAT-NOTES.md §5.11 (P1)** — `/api/events/:id` PATCH used to slip a `{url: null}` body past the route-layer superRefine because `youtubeUrlRequired` returned early when the body lacked `kind`. The row would then carry kind=youtube_video AND url=null after the UPDATE — a round-4 invariant violation. Plan 02.1-37 moves the validator into `updateEvent` (service layer): SELECT the existing row, compute `merged = {kind: input.kind ?? existing.kind, url: input.url !== undefined ? input.url : existing.url}`, then throw `AppError('kind_url_inconsistent', 422, {reason})` when `merged.kind === 'youtube_video'` and `merged.url` is falsy / non-YouTube. The route-layer schema drops `.superRefine(youtubeUrlRequired)` from `updateEventSchema` (comment block documents the move); `createEventSchema` keeps its superRefine (create body IS the full state — no partial-update problem).
- **Closed UAT-NOTES.md §5.13 (P2)** — `/feed` page loader cast `url.searchParams.getAll("kind")` directly into `FeedFilters["kind"]` so a malformed URL like `/feed?kind=foo` would let `foo` reach Drizzle's `inArray(events.kind, [...])` clause and surface as a Postgres 500. Plan 02.1-37 adds `src/lib/util/filter-event-kinds.ts` with `filterValidKinds(raw: string[]): EventKind[]` and routes the loader's kindList through it. Unknown values are silently dropped (matches the existing `?show=` malformed-param fallback at lines 87-95). VALID_EVENT_KINDS stays the single source of truth — service + route + page-loader all reference the same constant.
- **Privacy & multi-tenancy invariants preserved** — `updateEvent` keeps its existing userId-scoped row load; cross-tenant PATCH on a forged eventId surfaces as `NotFoundError → 404` BEFORE the merged-state validator runs. Test 4 in the new describe block asserts the response body never contains "forbidden" / "permission" / "kind_url_inconsistent" on cross-tenant access. AGENTS.md item 2 contract preserved.

## Decisions Made

- **Validator runs BEFORE `attachEventToGames`** in the new updateEvent ordering. The previous "gameIds first" comment is preserved with a Plan 02.1-37 footnote — the merged-state check goes earlier still so a kind/url inconsistency aborts cleanly without writing junction rows. Preserves "validate first; mutate after pass" (CLAUDE.md anti-pattern list, item 5).
- **`null` is a real intent to clear, distinct from `undefined`** — the conditional `input.url !== undefined ? input.url : existing.url` treats `null` as "set the url to null" so a PATCH `{url: null}` on a youtube_video row trips the validator. If we'd written `input.url ?? existing.url`, the null would silently fall through to the existing url and the bug would persist.
- **Error code = `'kind_url_inconsistent'`** (NOT `validation_failed`). More specific; metadata.reason discriminates between the two miss modes (`youtube_video_requires_url` vs `url_not_youtube`); clients can branch without parsing the human-readable message.
- **Existing-row SELECT includes `isNull(events.deletedAt)`** — matches the existing UPDATE WHERE clause exactly. Soft-deleted rows surface as 404 from updateEvent at the same point they always did.
- **Helper extracted to `src/lib/util/`** rather than inlined in the loader — unit-testable without mocking SvelteKit's URL object; reusable for any future loader / route that needs the same kindList defense.
- **EventKind imported from `data-source-adapter.js`** (the type's actual home) NOT from `dto.ts` (which doesn't export it). Plan's <interfaces> hinted at dto.ts; verified at execution time that data-source-adapter.ts is correct. Documented in helper comment.
- **Plan 02.1-17 Test 12 updated in-place** — its old assertion was correct against the OLD code path; the contract change is the documented behavior of Plan 02.1-37, so updating the test's expected shape is the right move (NOT an out-of-scope fix).
