# Phase 02.1 Plan 26: FeedQuickNav — chip strip at top of /feed Summary

**NEW <FeedQuickNav> component renders a chip strip / segmented control above DateRangeControl on /feed for single-click switching between the 4 most-common Show axis values (All / Inbox / Standalone / per-game with > 5 games collapsing to a 'More games ▾' dropdown). ## Decisions Made

- **Pure-component SSR testability via props-only context.** FeedQuickNav takes `currentUrlSearch: string` and `onNavigate: (href: string) => void` as props instead of importing `$app/state` / `$app/navigation` directly. The SSR test harness (audit-render.test.ts) renders Svelte 5 components via `svelte/server`'s `render(...)` which doesn't load SvelteKit virtual modules; importing `$app/state` would crash the test at component-import time. The parent `/feed/+page.svelte` threads `page.url.search` and `goto(href).then(() => invalidateAll())` into the component — three lines of wiring at the call site. This pattern mirrors Plan 02.1-21's schema-prop on FiltersSheet (consumer-pages-decide pattern over component-knows-its-context pattern) and is now established as the convention for any future component needing audit-render coverage.
- **URL contract consumer (NOT inventor).** FeedQuickNav consumes the existing `?show=any|inbox|standalone|specific&game=<id>` URL contract from Plans 02.1-19 + 02.1-24. NO new URL params, NO new endpoints, NO new audit verbs. The QuickNav is purely client-side URL navigation; the existing /feed loader's userId-first WHERE clause covers cross-tenant by construction (verified by the loader-contract guard in feed.test.ts). Privacy invariants 1, 6, 7 from CLAUDE.md preserved without any new code paths.
- **COLLAPSE_THRESHOLD = 5 games before overflow dropdown.** At 360px viewport with 4 fixed tabs (All / Inbox / Standalone / 'More games') + 5 game chips of typical indie title length (80-120px each), the strip is filled but each chip remains tappable. Dropping the threshold to 4 leaves dead space at common 6-7-game indie collections; promoting to 6+ delays the overflow ladder for less-common-case users. The choice is a UX guess that's load-bearing for round-3 closure but easy to tune later via a CSS container query when real users ship > 10 games.
- **Active-tab determination via TabKey discriminated union.** `"all" | "inbox" | "standalone" | { game: string }` — computed via `$derived.by` from `activeShow` (which the loader already builds from URL params via Plan 02.1-19 + 02.1-24's parsing). When `show.kind === 'specific' AND gameIds.length > 1` (FiltersSheet multi-select path), no quick-nav tab is highlighted — falls back to "all" (defensive); the FilterChips strip below still shows the multi-game chip so the user has visual context. The 0-element specific case (`gameIds.length === 0`) also falls back to "all".
- **buildHref preserves long-tail params, mutates only show / game / cursor.** Clicking a tab from a filtered view keeps date/kind/source/authorIsMe/all=1 applied; the QuickNav is a SHORTCUT to the Show axis, not a "reset every filter". Cursor is always cleared on tab change because the previous cursor pointed into the old result set. Implementation: `params.delete('cursor'); params.delete('show'); params.delete('game');` then set the new shape.
- **13 SSR render-time tests vs end-to-end browser tests.** The component-level contract (tab structure, active-state logic, href construction, overflow behavior, CSS overflow-x) is locked into the SSR test in audit-render.test.ts where it can run without a browser. The end-to-end browser flow (real click → URL change → loader re-runs → feed list refreshes) requires the cookie-injection auth harness still deferred to Phase 6. The 6 it.skip stubs in feed-360.test.ts (Plan 02.1-26 describe block) are grep-discoverable when the harness arrives. Same precedent as Plans 02.1-18 / 19 / 20 / 21 / 23.
