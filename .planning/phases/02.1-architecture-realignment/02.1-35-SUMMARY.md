## Decisions Made

- **Strip the entire `metadata.inbox` jsonb key on diff > 0** rather than `jsonb_set(metadata, '{inbox,dismissed}', 'false'::jsonb)`. The strip removes the inbox sub-object entirely when `dismissed` was the only entry, keeping the metadata object minimal. Cleaner forensics and equivalent behavior at the read site (`COALESCE(metadata->'inbox'->>'dismissed','false') = 'false'` returns true either way).
- **Audit writes OUTSIDE the transaction** — preserves AGENTS.md item 4 contract verbatim. A missing audit row on a successful business write is forensics-acceptable; an audit failure that rolls back the business write would block legitimate user actions on a forensics blip.
- **No new audit verbs for the metadata clear** — the existing `event.attached_to_game` / `event.detached_from_game` rows are enough forensic context per UAT-NOTES.md §5.1 'Suggested fix'. The metadata mutation is implicit re-engagement, not an explicit user action.
- **No-op calls preserve metadata** — `diffNonEmpty=false` (toAdd=[] AND toRemove=[]) takes the second branch of the conditional and only bumps `updatedAt`. This preserves the existing contract for redundant attachEventToGames calls (e.g. UI submitting the same gameIds set the event already has).
- **Re-SELECT the parent row after the transaction** — Drizzle's `.returning()` on the UPDATE inside the transaction wasn't viable because the metadata column is computed via the `sql` template; the re-SELECT outside the transaction reads the post-strip shape so the EventDto projection downstream sees the cleared inbox key.
- **Rollback test mechanism: Proxy-wrapped tx parameter via `vi.spyOn(db, 'transaction')`** — Drizzle's `tx` object has a different prototype than `db`, so spying on `db.insert` doesn't intercept the inner `tx.insert` calls. Wrapping `db.transaction` itself lets us proxy the tx parameter and inject a synthetic throw inside the real Postgres transaction. The thrown error propagates out, the real transaction ROLLBACK fires, and we observe the rollback at the row level. This pattern is now available for future rollback assertions in the codebase.

## Deviations from Plan

The plan's acceptance criteria included `grep -c "tx.insert(events)"`, `grep -c "tx.delete(eventGames)"`, and `grep -c "tx.update(events)"` returning >= 1. Drizzle's chained API splits `tx.<verb>(table).<modifier>(...)` across multiple lines for readability (e.g. `await tx\n   .delete(eventGames)\n   .where(...)`). The literal greps don't match the multi-line shape, but the calls are present and the behavioral tests assert the actual rollback / atomicity contract. No code change made to satisfy the literal grep — the multi-line style matches the codebase convention (see `games.ts:243-260` for the precedent).
