## Accomplishments

- New `services/data-sources.ts` (~340 lines) ships seven exported functions: `createSource`, `listSources`, `getSourceById`, `updateSource`, `softDeleteSource`, `restoreSource`, `findSourceByAuthorUrl`. Every function takes `userId: string` first; every Drizzle query carries `eq(dataSources.userId, userId)`.
- Non-functional source kinds (`reddit_account` / `twitter_account` / `telegram_channel` / `discord_server`) reject with `AppError('kind_not_yet_functional', 422)` and metadata `{ kind, available_phase }` per RESEARCH §5.4 — no orphan rows can land for kinds Phase 3+ adapters don't yet support.
- Postgres 23505 unique-violation on the partial unique index `data_sources_user_handle_active_unq` translates to `AppError('duplicate_source', 422, { handle_url })`. The error walks `DrizzleQueryError.cause` to find the SQLSTATE — drizzle-orm 0.45's wrapper would otherwise hide the code from a bare `'code' in err` check.
- Soft-delete sets `deleted_at` and the `source.removed` audit row writes BEFORE the UPDATE (D-32 forensics — Phase 2 `removeSteamKey` precedent). Idempotent on the second call: a soft-deleted row throws NotFoundError on a follow-up softDelete attempt.
- Restore clears `deleted_at` only when within `env.RETENTION_DAYS` of the deletion; older deletions throw `AppError('retention_expired', 422)`. Cross-tenant restore throws NotFoundError.
- `updateSource` audits `source.toggled_auto_import` ONLY when `autoImport` actually changes value — display-name renames carry no audit verb (D-32 closed picklist).
- `findSourceByAuthorUrl` is the seam Plan 02.1-05's events service consumes for INGEST-03 author_is_me inheritance: matches active `youtube_channel` data_sources where `is_owned_by_me=true` and `handle_url = authorUrl`.
- `dto.ts` ships `toDataSourceDto` (replaces `toYoutubeChannelDto`) with explicit field listing — `userId` stripped at runtime even when an input row literal carries it; behavioural test in `tests/unit/dto.test.ts` enforces.
- 18 live integration tests in `tests/integration/data-sources.test.ts` cover the full SOURCES-01 + SOURCES-02 surface: happy path, kind gating (4 kinds), validation, dedup, audit metadata, listSources active-only filter, cross-tenant 404 (3 paths), soft-delete idempotency, restore + retention boundary, audit-before-update ordering (vi.spyOn writeAudit), updateSource toggle audit emission rules, soft-delete-allows-resurrect via partial unique index.
- The Phase 2 `services/youtube-channels.ts` is deleted entirely (`git rm`); responsibilities migrate to `data-sources.ts` (CRUD) + Plan 02.1-05's events service (per-event game attach replaces the channel-level attach).

## Audit ordering decision (D-32)

`softDeleteSource` writes `source.removed` BEFORE the soft-delete UPDATE — even if the UPDATE fails, the security signal is captured. Mirrors Phase 2's `removeSteamKey` precedent. The integration test (`Plan 02.1-04: softDeleteSource writes audit BEFORE the soft-delete update`) spies on `writeAudit`; inside the spy callback it queries `data_sources` and asserts `deleted_at IS NULL` at the moment the audit fires. The test would fail if a future refactor flips the order.

## Decisions Made

- **`toEventDto` full extension deferred to Plan 02.1-05 per parallel-execution scope.** The plan's `<action>` Step C said "UPDATE EventDto + toEventDto per RESEARCH §10.2" — but the orchestrator's parallel-execution block explicitly carves out: "If you discover that dto.ts also needs an updated toEventDto for Plan 05's events shape, leave that work for Plan 05 — only touch the toDataSourceDto projection here." The parallel-execution block is the authoritative override. The minimum-compile fix here (kind union widening + gameId nullable) is necessary because Plan 02.1-01 changed the schema and `dto.ts` would otherwise fail tsc — the rest of RESEARCH §10.2 is Plan 05's scope.
- **`toYoutubeVideoDto` removed entirely.** No service caller remains in Phase 2.1 (the `tracked_youtube_videos` schema was retired in Plan 02.1-01 and the `services/items-youtube.ts` service is being deleted by Plan 02.1-05). Leaving the projection in dto.ts as dead code would be a maintenance drag and would obscure the unified-events-table design intent.
- **`isPgUniqueViolation` walks `DrizzleQueryError.cause`.** drizzle-orm 0.45 wraps the underlying pg error in `DrizzleQueryError`. A bare `'code' in err && err.code === '23505'` check (as the plan's `<action>` skeleton showed) is a vacuous-pass against this DB layer — `DrizzleQueryError` has no `.code` property; the actual error is on `.cause`. The behavioural duplicate test (`Plan 02.1-04: duplicate translates PG 23505`) caught this on first run and drove the fix. Bounded to depth=5 to avoid infinite loops on cyclic causes.
- **`restoreSource` does NOT write an audit row.** RESEARCH §4.4 reserves the `audit_action` enum for security-relevant operations; restoration of a user's own resource is a low-risk operator action. The function takes `ipAddress` / `userAgent` parameters with `_`-prefixed names so the call-site signature is forward-compatible if a future `source.restored` verb is added (Phase 6 housekeeping per RESEARCH §4.4).
- **`softDeleteSource` returns the soft-deleted row** (not void). Wave 2's `DELETE /api/sources/:id` route can render the retention badge from the response without a follow-up GET — saves a roundtrip on the most common UX flow.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `isPgUniqueViolation` against drizzle-orm 0.45's `DrizzleQueryError` wrapper**
- **Found during:** Task 1 GREEN verification (the duplicate test failed with the wrapper revealed in the assertion diff).
- **Issue:** The plan's `<action>` skeleton showed `isPgUniqueViolation` as a one-liner against `err.code`. drizzle-orm 0.45 wraps the pg error in `DrizzleQueryError` whose `.code` is undefined; the SQLSTATE lives on `.cause.code`. Without the fix the duplicate-source translation never fires and the integration test surfaces a raw `DrizzleQueryError` instead of the expected `AppError('duplicate_source', 422)`.
- **Fix:** Walk the `.cause` chain to a bounded depth (5) and check each link for `code === '23505'`. Mirrors the items-youtube precedent adapted for the wrapper.
- **Files modified:** `src/lib/server/services/data-sources.ts`.
- **Commit:** `3127840` (folded into the GREEN commit since the plan's skeleton was a sketch, not a fixture).
