## Decisions Made

- **Path B (unconditional DB constraint + pre-INSERT lookup + race-window catch) chosen over Path A (partial-WHERE constraint).** Plan 02.1-27 had already shipped the constraint as UNCONDITIONAL `unique(game_id, app_id)`. Path A would require dropping that constraint and adding `unique(game_id, app_id) WHERE deleted_at IS NULL` — a destructive migration on top of Plan 27's just-shipped migration, AND it would silently allow re-adding soft-deleted same-game rows (breaking the audit forensic chain). Path B keeps the constraint shape, catches soft-deleted dupes via the pre-INSERT lookup (returning existingState='soft_deleted' so the UI knows to offer Restore), and uses the catch around the INSERT only as a race-window backstop. This matches the Plan 02.1-14 events soft-delete-Restore precedent (the Restore-only UX path).
- **Pre-INSERT lookup is gameId-scoped (NOT app-id-scoped).** Cross-game same-appId is ALLOWED post-Plan-27 — the user-scoped unique constraint is gone, so the same Steam appId can attach to multiple games of the same user (e.g., a Portal-2 main card + a Portal-2 review-notes card). The pre-INSERT lookup uses `eq(userId) AND eq(gameId) AND eq(appId)` so cross-game same-appId attempts fall through to the INSERT (post-27 expected behavior); only same-tenant same-game existing rows short-circuit. This preserves the P19-style by-construction tenant isolation: userB's appId never surfaces in userA's existingGameId payload.
- **isPgUniqueViolation extracted to src/lib/server/db/postgres-errors.ts.** The data-sources.ts version walked the cause chain (DrizzleQueryError wrapping); the events.ts version was a shallow check. Plan 02.1-29 needed the cause-chain walker (drizzle-orm wraps the pg error). Extracting to a shared module prevents duplication and keeps the depth=5 bound consistent. data-sources.ts updated to import; the local function deleted; behavior verified identical (no test changes needed in data-sources.test.ts beyond the existing 'Plan 02.1-04: duplicate (user_id, handle_url) translates PG 23505 into duplicate_source (422)' which exercises the same code path through the new import). The events.ts shallow variant stays separate — different call site, different error shape, no DrizzleQueryError wrapping.
- **mapErr extended to forward AppError.metadata into the JSON body when non-empty.** The current contract was `{error: code}` only; UAT-NOTES.md §4.25.G requires `existingGameId` + `existingState` on the wire so Plan 02.1-30 UI can render an actionable toast (e.g., "Portal 2 is already attached to game X — open?" for active duplicates; "Portal 2 is in your removed listings — restore?" for soft-deleted). Backward compat: callers without metadata (most existing AppError throws) see the same `{error: code}` body. The `Object.keys(err.metadata).length > 0` guard keeps the wire minimal for those callers.
- **Cross-tenant DELETE probe (§4.25.H test surface) verified pre-existing.** tests/integration/tenant-scope.test.ts already includes the DELETE listings probe at lines 322-324 (Plan 02-08 D-37 matrix). The plan still adds Plan 02.1-29 Test 5 in game-listings.test.ts to own the test surface for §4.25.H closure (per the plan's truth #5: "Plan 29 owns the verification surface"). The DELETE handler itself was shipped by Plan 02-08 at lines 69-77 of routes/game-listings.ts (verified via grep at task time, returns ≥1 match — VERIFY path taken, ADD path not needed).
- **Test for soft-deleted Path B uses direct UPDATE rather than removeSteamListing.** Test 2 soft-deletes the row by directly UPDATEing `deletedAt = new Date()` (rather than calling `removeSteamListing`). This keeps Test 2 focused on the addSteamListing pre-INSERT lookup contract — the test asserts that a row with `deletedAt IS NOT NULL` triggers existingState='soft_deleted'. Going through removeSteamListing would couple Test 2 to the removeSteamListing behavior; the direct UPDATE is the load-bearing path the plan's truth #2 documents.

## Patterns Established

- **Path B pattern (pre-INSERT lookup + race-window catch + unconditional DB constraint).** For any future service where soft-delete-Restore is the intended UX AND the DB has an unconditional unique constraint on the resource shape, follow this shape: defensive same-tenant lookup BEFORE INSERT (resource-scoped, no `isNull(deletedAt)` filter) returns existingState=active|soft_deleted → AppError 422 with metadata; then a try/catch around the INSERT translates the race-window 23505 to existingState='active'. The metadata payload reaches the wire via mapErr → JSON body for the UI to render structured hints. Plan 02.1-14 events soft-delete-Restore is the original precedent; Plan 02.1-29 codifies it for the listings table.
- **Shared `db/postgres-errors.ts` module.** For any future Postgres-error predicate that more than one service needs, add to the shared module (naming convention: `isPg<Code>Violation(err: unknown, depth = 5): boolean`). Extend by appending new predicates rather than spreading them across services. The depth=5 bound is the project default for cause-chain walks.
