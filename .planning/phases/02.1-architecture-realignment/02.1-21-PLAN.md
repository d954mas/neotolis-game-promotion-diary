<objective>
Generalize FiltersSheet + FilterChips with an explicit `schema` prop so each consumer page (/feed, /audit) controls WHICH filter axes render. Adds the missing date-range filter to /audit (UAT-NOTES.md §9.2-bug user quote: "В окне аудита нет возможности выбрать дату как в feed"). Removes the irrelevant /feed-only fields (kind / source / show / authorIsMe) that bled into /audit's FiltersSheet via Plan 02.1-20's shoehorn (UAT-NOTES.md §9.2-bug user quote: "Открывает окно, там есть типы аудита, но и куча полей из feed фильтров").

Purpose: This is a Wave 0 foundation plan. Both /feed and /audit consume the new schema-prop API; downstream round-3 plans (22, 23, 24, 25, 26) all rely on the FiltersSheet/FilterChips contract being clean before they touch related surfaces. Plan 02.1-20 introduced the regression by adding `filters.action` as an opt-in flag; the right shape is an explicit schema array.

Output:
- FiltersSheet.svelte gains `schema` prop; per-axis fieldset rendering switches from implicit detection to explicit schema-driven loop.
- FilterChips.svelte gains `schema` prop; chip emission honors it.
- /feed passes the full schema (preserves current behavior).
- /audit passes only ['action', 'date']; gains a working date-range filter.
- listAuditPage(userId, cursor, actionFilter[], dateRange?) signature widens to accept date range; userId clause stays FIRST.
- Test coverage: integration regression guards (audit-render.test.ts) + browser test (/audit at 360px).
- Paraglide invariant honored: any new keys added with EXPECTED_KEYS update in same commit.
</objective>
