## Accomplishments

- **§4.22.B closed** — Remove button hidden in read-mode; the read-mode `.actions` row now hosts ONLY the Edit pencil. Remove moved into the edit-form footer where /events parity (Plan 02.1-32) keeps destructive actions confined to the edit surface.
- **§4.22.C closed** — Edit pencil hidden when `editing === true`. The `{#if !editing}` ... `{:else}` ... `{/if}` structure encodes the visibility gate; no duplicate pencil-to-enter-edit-mode affordance once the user is already editing.
- **§4.22.D closed (with regression guard)** — Plan-time review confirmed the "duplicate text input" finding was stale (no parallel control existed on this branch). Plan 02.1-33 keeps the EXACT current shape (ONE `<input type="checkbox" bind:checked={editAutoImport}>`) and adds three-layer negative-grep regression assertions: no `type="text"` references editAutoImport, no `bind:value={editAutoImport}`, no `type="text"` whose attribute name mentions auto-import in any casing/separator.
- **§4.22.E closed** — Save / Cancel / Remove sit at the BOTTOM of the edit-form block, separated from form fields by a `<hr class="section-divider" />`. The form-footer is a flex row with Save (`footer-btn-primary`) + Cancel (`footer-btn-ghost`) + Remove (`footer-btn-danger`) at full-width on 360px and at flex-grow on 768px+. Closes user quote: "Кнопки save cancel нужно внизу карточки делать, иначе не очевидно где они и зачем".

## Decisions Made

- **common_save vs toast_saved** — Previous Save button reused `m.toast_saved()` ("Saved.") which is the post-action toast text, a subtly wrong label for a button that has not yet acted. Added `common_save: "Save"` to messages/en.json (already absorbed in HEAD via Plan 02.1-30 commit) and updated EXPECTED_KEYS.
- **§4.22.D regression-prevention strategy** — Plan-time review showed the "duplicate text input" did not exist on this branch (likely stale from a pre-Plan-02.1-22 iteration). Rather than remove nonexistent UI, Plan 02.1-33 keeps the current single-checkbox shape AND adds 3-layer negative-grep regression guards (positive-count + type-text + value-bind + casing catch-all) so any future regression to a parallel string control is caught at SSR scan time.
- **Browser-mode test deferred to Phase 6** — The full interactive end-to-end (click Edit → form expands → click Cancel → returns to read-mode) requires an authenticated `/sources` route which vitest-browser cannot mint without a Postgres harness. Stub-skipped placeholders carry the manual-UAT recipe; SSR-level regression guards in audit-render.test.ts are the load-bearing protection. Same precedent as Plans 02.1-18 / 19 / 20 / 21 / 22 / 23 / 24 / 25 / 26 / 28 / 30 / 32 / 34.
- **Preserved Plan 02.1-30's `var(--color-mine)` token** — When merging my Plan 02.1-33 changes onto a SourceRow.svelte already updated by Plan 02.1-30, kept the `--color-mine` token usage on `.row.mine` and `.ownership-badge.mine` rather than reverting to `--color-accent`. The two failing audit-render tests that assert `var(--color-accent)` are Plan 02.1-30's test-update gap, not a Plan 02.1-33-introduced regression — logged to deferred-items.md.
