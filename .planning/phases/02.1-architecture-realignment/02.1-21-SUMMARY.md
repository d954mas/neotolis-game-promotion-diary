## Decisions Made

- **Schema prop is REQUIRED (no default).** Consumer pages must opt in to each axis explicitly. This eliminates the Plan 02.1-20 implicit gate and prevents leak regressions by construction. A future Phase 3+ axis addition only requires passing a different schema array.
- **'date' axis present in both schemas but emits no chip.** The visible `<DateRangeControl>` from/to inputs ARE the indicator; duplication via a chip is confusing (Plan 02.1-19 contract preserved). The 'date' entry in schema gates only fieldset rendering inside the sheet.
- **/audit deliberately omits the 30-day default window.** Auditing is investigative; "show me everything" is the right default. /feed's UX choice (30-day window for daily workflow) doesn't translate.
- **Sheet's in-sheet date emits the same { from, to } payload shape as <DateRangeControl>.** Consistent contract across both date-axis entries; consumer maps the same shape to URL params on apply.
- **listAuditPage dateRange parameter is optional with default undefined.** Existing callers (Plan 02.1-20 audit loader before this plan, integration tests passing only the action filter) keep working. New callers pass `{ from?, to? }` or omit entirely.

- **Issue:** The plan's literal-grep acceptance criterion (`grep "schema={\["`) suggests inline literal arrays. **2. [Rule 2 - Missing critical] applyFiltersFromSheet on /feed honors sheet-emitted from/to**
- **Found during:** Task 2 (wiring /feed page)
