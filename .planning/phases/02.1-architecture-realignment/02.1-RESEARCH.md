# Phase 2.1: Architecture Realignment - Research

**Researched:** 2026-04-28
**Domain:** Drizzle/Postgres schema migration (rename + extend + drop), unified service layer refactor, SvelteKit 3-view IA build (`/feed` + `/sources` + `/games/[id]` rebuild), forward-compat data-source adapter interface.
**Confidence:** HIGH (locked stack, zero production data, established codebase patterns ship cleanly to the realignment) — with one MEDIUM caveat: drizzle-kit 0.31 does NOT auto-emit `ALTER TYPE ... RENAME VALUE` for the audit_action rename (§4 — verified via [drizzle-orm#5340](https://github.com/drizzle-team/drizzle-orm/issues/5340)) so the planner MUST budget for a hand-written companion migration.

## Summary

Phase 2.1 is a structural realignment of Phase 2's data and IA: rename `youtube_channels` → `data_sources`, drop `tracked_youtube_videos`, extend `events` with `kind`/`author_is_me`/`source_id`/nullable `game_id`, rename audit enum values, refactor services + routes, build three SvelteKit pages (`/feed` primary, `/sources`, `/games/[id]` rebuild), bundle UI polish, extend smoke. **Zero production data** (Phase 2 is dev-only) so the migration is destructive but cheap.

**Three load-bearing findings the planner MUST internalize:**

1. **drizzle-kit 0.31 cannot rename enum values.** Verified via the project's pinned `drizzle-kit ^0.31.0` against the upstream issue [drizzle-orm#5340](https://github.com/drizzle-team/drizzle-orm/issues/5340) — drizzle-kit emits a broken USING-cast pattern (cast-to-text → DROP TYPE → CREATE TYPE → cast-back-with-USING) that fails when existing rows carry the old values. **The audit_action rename (`channel.* → source.*`, `item.* → event.*`) requires a hand-written companion migration using the Postgres-native `ALTER TYPE ... RENAME VALUE` syntax** ([Postgres 16 docs](https://www.postgresql.org/docs/16/sql-altertype.html)). See §4.
2. **drizzle-kit's `RENAME TABLE` interactive prompt is not viable in CI.** Renames are detected via interactive CLI ("did you rename X to Y? Y/N"). The plan must run `drizzle-kit generate` once locally with the rename answered explicitly, then commit the resulting SQL — the CI `pnpm db:generate` path stays a check, not a writer. See §1.
3. **The full set of changes is forward-only and atomic-friendly.** Rename + add columns + drop table + extend `events` all live in a single migration file because Postgres DDL is transactional and the migration runs under the existing advisory lock at boot (Phase 1 Plan 03 pattern). The audit-enum hand-written companion migration is a SEPARATE file that runs immediately after, also under the same lock. See §9.

**Primary recommendation:** Land the schema migration as `drizzle/0002_phase02_1_data_sources.sql` (auto-generated for table rename + column adds + DROP) plus `drizzle/0003_phase02_1_audit_enum_rename.sql` (custom, `ALTER TYPE ... RENAME VALUE` + `INSERT INTO drizzle.__drizzle_migrations` parity), in that order. Follow with the service refactor under the unified `events` model, then the route refactor, then the three SvelteKit pages, then UI polish, then smoke extension.

1. **Tenant scoping is mandatory and explicit** — every service function takes `userId: string` as first non-optional argument; every Drizzle query against a user-owned table includes `eq(<table>.userId, userId)`. 4. **Audit log INSERT-only and tenant-relative** — `writeAudit` only path; `(user_id, created_at desc)` cursor. Application DB role lacks UPDATE/DELETE on `audit_log`. Audit enum rename MUST be a forward-only `ALTER TYPE RENAME VALUE` operation (no `UPDATE` rows; see §4).
5. **DTO discipline strips secrets at projection layer** — every entity has a `to<Entity>Dto` projection in `src/lib/server/dto.ts`. Phase 2.1 adds `toDataSourceDto` (replaces `toYoutubeChannelDto`) and updates `toEventDto`; behavioral test in `tests/unit/dto.test.ts` asserts strip happens at runtime (§10).
6. **Pino redact paths** — Phase 2.1 introduces no new secret-shaped fields. `data_sources.metadata` jsonb is non-secret per-platform config (e.g. `uploads_playlist_id`); if a future kind needs a secret in metadata, it lives in `api_keys_*` instead, not in `data_sources.metadata`.

## 1. Migration Mechanics

**Confidence: HIGH** (Postgres SQL operations are unambiguous; drizzle-kit 0.31 emission verified via the project's existing `0001_phase02_schema.sql`).

### 1.1 Confirmed safe SQL operations

All four operations the orchestrator's scope demands are supported by Postgres 16 (locked stack target) and run inside a single transaction. Forward-only — no `DOWN` migration.

#### (a) RENAME TABLE `youtube_channels` → `data_sources` preserving FKs

```sql
-- Postgres-native; transactional; preserves PK, indexes (auto-renamed by Postgres),
-- and FK targeting from game_youtube_channels.channel_id REFERENCES youtube_channels(id).
-- The FK constraint name itself is NOT auto-renamed — but it doesn't need to be;
-- "youtube_channels_user_id_fkey" still pointing at "data_sources" by OID is valid.
-- We rename it for hygiene.
ALTER TABLE "youtube_channels" RENAME TO "data_sources";
ALTER INDEX "youtube_channels_user_id_idx" RENAME TO "data_sources_user_id_idx";
ALTER INDEX "youtube_channels_user_handle_unq" RENAME TO "data_sources_user_handle_unq";
-- FK constraint hygiene rename (optional but kept for diff legibility):
ALTER TABLE "data_sources" RENAME CONSTRAINT "youtube_channels_user_id_user_id_fk" TO "data_sources_user_id_user_id_fk";
```

The M:N link table `game_youtube_channels` is **dropped entirely** in this migration — Phase 2.1 retires the per-game channel attachment in favor of per-event `game_id` attachment (FEED-01 / GAMES-04a reframed). Plan's call: drop it; the audit log already captured the historical `channel.attached` / `channel.detached` events; the events themselves carry the per-event `game_id` going forward.

```sql

```

### 1.3 The interactive-prompt gotcha (CI implications)

`drizzle-kit generate` uses an **interactive CLI prompt** to disambiguate "did you rename X to Y, or drop X and create Y?" ([source](https://orm.drizzle.team/docs/migrations)). This means:

- **Locally**: developer runs `pnpm db:generate`, drizzle-kit asks "rename youtube_channels to data_sources? [Y/N]", developer answers Y, file lands in `drizzle/`.
- **In CI**: `pnpm db:check` is the diff-detector (runs non-interactively), but `pnpm db:generate` is NOT in CI. The committed SQL file IS the contract. CI runs `drizzle-kit migrate` (not generate) which is fully non-interactive and just executes the committed SQL.

Plan implication: the migration generation is a **local one-shot** by the executing agent, captured in a single commit. CI never re-generates. This matches the project's existing pattern (Phase 2 generated `0001_phase02_schema.sql` once and committed it).

### 1.4 Transaction semantics + advisory lock

The Phase 1 `runMigrations()` runner already wraps every migration file in a transaction and holds advisory lock `0x4D49475241544531` so multiple replicas don't race (STATE.md decision recorded). Phase 2.1's two migration files inherit this for free.

**One caveat:** `ALTER TYPE ... ADD VALUE` in Postgres 12+ inside a transaction makes the new value unusable **until the transaction commits**. This is fine for Phase 2.1 because:
- The migration file containing `ALTER TYPE event_kind ADD VALUE 'youtube_video'` does NOT itself try to insert any rows with `kind = 'youtube_video'` — that happens in service-layer code at runtime, AFTER the migration commits.
- The audit_action `RENAME VALUE` migration (§4) does not have this restriction — `RENAME` is fully transactional with no usage delay.

## 2. DataSourceAdapter Interface — Locked Shape

**Confidence: HIGH** for the interface shape; **MEDIUM** for the specific TypeScript signatures (the planner has discretion on `Promise<RawEvent[]>` vs `AsyncIterable<RawEvent>` — recommendation locked below based on Phase 3 quota math).

### 2.1 Locked TypeScript shape (Phase 2.1 ships this; Phase 3 implements pollContent / pollStats)

```typescript
// src/lib/server/integrations/data-source-adapter.ts (NEW — Phase 2.1)

import type { dataSources, events } from "../db/schema/index.js";

export type DataSourceRow = typeof dataSources.$inferSelect;
export type EventRow = typeof events.$inferSelect;

export type SourceKind =
  | "youtube_channel"
  | "reddit_account"
  | "twitter_account"
  | "telegram_channel"
  | "discord_server";

/**
 * RawEvent — what an adapter returns from pollContent before the events service
 * persists it. Note `author_is_me` is set by the SERVICE LAYER (inherited from
 * the source.is_owned_by_me) — the adapter just discovers content. Per-kind
 * metadata jsonb is the adapter's responsibility (e.g. youtube_channel sets
 * { view_count, like_count, comment_count, last_polled_at }).
 */
export interface RawEvent {
  externalId: string;          // adapter-specific (e.g. youtube video_id, reddit post id)
  url: string;                 // canonical event URL
  title: string;               // user-facing title
  occurredAt: Date;            // when the content was published (NOT poll time)
  kind: EventKind;             // closed picklist matching events.kind enum
  metadata: Record<string, unknown>; // per-kind jsonb shape — see §10 for kind-by-kind
}

/**
 * StatsSnapshot — what an adapter returns from pollStats. Phase 3's polling
 * worker writes one row of (event_id, polled_at, metric_key, metric_value)
 * per metric returned (the table is event_stats_snapshots — Phase 3 lands it).
 * Phase 2.1 only locks the type shape; no consumer until Phase 3.
 */
export interface StatsSnapshot {
  polledAt: Date;
  metrics: Record<string, number>;  // e.g. { view_count: 1234, like_count: 56, comment_count: 7 }
  status: "ok" | "rate_limited" | "auth_error" | "not_found" | "private";
}

/**
 * DataSourceAdapter — one per source_kind. Phase 2.1 ships only the
 * youtube_channel adapter STUB (the interface + a thin file with `pollContent
 * not implemented` / `pollStats not implemented` throws). Phase 3 fills in
 * the implementations.
 *
 * Design discipline (from PROJECT.md "Architecture"):
 *   - Pure adapters: NO db writes, NO logging side effects, NO global state.
 *   - The events service is the ONLY place that decides author_is_me /
 *     source_id / inserts the row. Adapter just discovers + reports.
 *   - Dedup is the EVENTS SERVICE's job (UNIQUE(user_id, kind, source_id, external_id)
 *     where source_id IS NOT NULL — see §1).
 */
export interface DataSourceAdapter {
  readonly kind: SourceKind;

  /**
   * Discover new content from a registered data_source since `since`.
   * Returns Promise<RawEvent[]> not AsyncIterable — quota math (todo PROMOTED:
   * 1 unit per 50-video page) means each pollContent call retrieves at most one
   * page (50 items); a backlog import is a separate "import last N=100 videos"
   * one-shot operation that's outside the worker loop. Array fits.
   *
   * Idempotency: caller MUST pass a `since` derived from the row's
   * `data_source.metadata->>'last_polled_at'`; the adapter trusts that bound
   * and does not deduplicate against the database itself (events service does).
   */
  pollContent(source: DataSourceRow, since: Date): Promise<RawEvent[]>;

  /**
   * Fetch latest stats for a single event. Phase 3 worker writes the snapshot
   * to event_stats_snapshots. Returns status='private'/'not_found'/'auth_error'
   * with empty metrics when the underlying content disappeared or auth broke;
   * caller writes the status to events.last_poll_status.
   */
  pollStats(event: EventRow, source: DataSourceRow | null): Promise<StatsSnapshot>;
}

/**
 * Adapter registry — one entry per kind. Phase 2.1 ships only the youtube_channel
 * stub; Phase 3 fills in the implementations.
 */
export type EventKind =
  | "youtube_video"
  | "twitter_post"
  | "telegram_post"
  | "discord_drop"
  | "reddit_post"
  | "conference"
  | "talk"
  | "press"
  | "other";
```

**Locked:** identical pattern to `src/lib/server/services/audit-read.ts` — `base64url(JSON.stringify({at: ISO, id}))` of `(occurredAt, id)`.
```typescript
// src/lib/server/services/events.ts (NEW listFeedPage)

import { encodeCursor, decodeCursor } from "./audit-read.js"; // REUSE — same pattern

export const FEED_PAGE_SIZE = 50;

export interface FeedFilters {
  source?: string;
  kind?: EventKind;
  game?: string;
  attached?: boolean;          // null/undefined = no filter
  authorIsMe?: boolean;
  from?: Date;
  to?: Date;
}

export interface FeedPage {
  rows: EventRow[];
  nextCursor: string | null;
}

export async function listFeedPage(
  userId: string,
  filters: FeedFilters,
  cursor: string | null,
): Promise<FeedPage> {
  let parsedCursor: { at: Date; id: string } | null = null;
  if (cursor) parsedCursor = decodeCursor(cursor);

  const cursorClause = parsedCursor
    ? sql`(${events.occurredAt}, ${events.id}) < (${parsedCursor.at}, ${parsedCursor.id})`
    : sql`true`;

  const whereParts = [
    eq(events.userId, userId),                                       // P1: userId FIRST
    isNull(events.deletedAt),
    cursorClause,
  ];

  if (filters.source !== undefined) whereParts.push(eq(events.sourceId, filters.source));
  if (filters.kind !== undefined) whereParts.push(eq(events.kind, filters.kind));
  if (filters.game !== undefined) whereParts.push(eq(events.gameId, filters.game));
  if (filters.attached === true) whereParts.push(isNotNull(events.gameId));
  if (filters.attached === false) {
    whereParts.push(isNull(events.gameId));
    // Inbox view: also exclude inbox_dismissed=true (§6).
    whereParts.push(sql`COALESCE(${events.metadata}->>'inbox_dismissed', 'false') = 'false'`);
  }
  if (filters.authorIsMe !== undefined) whereParts.push(eq(events.authorIsMe, filters.authorIsMe));
  if (filters.from !== undefined) whereParts.push(gte(events.occurredAt, filters.from));
  if (filters.to !== undefined) whereParts.push(lte(events.occurredAt, filters.to));

  const rows = await db
    .select()
    .from(events)
    .where(and(...whereParts))
    .orderBy(sql`${events.occurredAt} DESC, ${events.id} DESC`)
    .limit(FEED_PAGE_SIZE + 1);

  const hasMore = rows.length > FEED_PAGE_SIZE;
  const page = rows.slice(0, FEED_PAGE_SIZE);
  const last = page[page.length - 1];
  return {
    rows: page,
    nextCursor: hasMore && last ? encodeCursor(last.occurredAt, last.id) : null,
  };
}
```

## 4. Audit Enum Migration

**Confidence: HIGH** (Postgres 12+ `ALTER TYPE ... RENAME VALUE` is fully transactional + drizzle-kit limitation is verified). **THIS IS THE LOAD-BEARING FINDING OF THE WHOLE RESEARCH** — the planner MUST split the migration into two files because of it.

### 4.1 The drizzle-kit gap

[drizzle-orm#5340](https://github.com/drizzle-team/drizzle-orm/issues/5340) confirms: when an enum value is renamed in `pgEnum(...)` declaration, drizzle-kit 0.31 generates this broken sequence:

```sql
-- drizzle-kit 0.31 EMITS (BROKEN — fails on existing data):
ALTER TABLE "audit_log" ALTER COLUMN "action" SET DATA TYPE text;
DROP TYPE "audit_action";
CREATE TYPE "audit_action" AS ENUM (...new values...);
ALTER TABLE "audit_log" ALTER COLUMN "action" SET DATA TYPE "audit_action" USING "action"::"audit_action";
-- ☝ FAILS HERE because old text values 'channel.added' don't exist in the new enum.
```

The Postgres-native fix is `ALTER TYPE ... RENAME VALUE` — supported since Postgres 10, fully transactional (verified via [Postgres 16 ALTER TYPE docs](https://www.postgresql.org/docs/16/sql-altertype.html)), no usage-restriction (unlike `ALTER TYPE ADD VALUE`).

### 4.2 Locked strategy: hand-write companion migration

```sql
-- drizzle/0003_phase02_1_audit_enum_rename.sql (HAND-WRITTEN companion)
-- Phase 2.1 audit_action vocabulary alignment — channel.* → source.*, item.* folded into event.*.
-- drizzle-kit 0.31 cannot emit ALTER TYPE RENAME VALUE (drizzle-orm#5340); this file is the
-- forward-only companion.
--
-- Postgres-native; transactional in Postgres 16; no usage-after-commit restriction
-- (unlike ALTER TYPE ADD VALUE).

-- Phase 2 channel.* values (none currently emitted by audit writers — channel CRUD has no
-- audit verb per the youtube-channels.ts file header comment "NO audit entries from this
-- service"). So the rename is a name-only operation; no UPDATE rows needed.
-- BUT we still rename them so the AUDIT_ACTIONS const in src/lib/server/audit/actions.ts
-- can be source-of-truth-aligned.
ALTER TYPE "public"."audit_action" RENAME VALUE 'channel.added' TO 'source.added';
ALTER TYPE "public"."audit_action" RENAME VALUE 'channel.removed' TO 'source.removed';
ALTER TYPE "public"."audit_action" RENAME VALUE 'channel.attached' TO 'source.attached_to_game';
ALTER TYPE "public"."audit_action" RENAME VALUE 'channel.detached' TO 'source.detached_from_game';

-- item.* folds into event.* (the items-youtube service IS gone in 2.1, and existing rows
-- in dev DB carry item.created / item.deleted — these RENAME without data UPDATE because
-- both names are valid "the same logical operation just with new vocabulary").
ALTER TYPE "public"."audit_action" RENAME VALUE 'item.created' TO 'event.created_from_paste';
ALTER TYPE "public"."audit_action" RENAME VALUE 'item.deleted' TO 'event.deleted_from_paste_origin';
-- Wait — these COLLIDE with existing event.created / event.deleted values. Do NOT RENAME;
-- instead drop the item.* values and UPDATE rows in dev DB. See §4.3 for the chosen path.
```

### 4.3 Wait — the collision problem

The orchestrator's scope says "fold `item.*` into `event.*`". Drilling in: AUDIT_ACTIONS already contains `'event.created'`, `'event.edited'`, `'event.deleted'` (from Phase 2). Renaming `'item.created'` → `'event.created'` would collide with the existing value. Postgres errors: "value already exists in enum".

**Two viable paths:**

**Path A — rename to a NEW vocabulary** (no collision):
```sql
ALTER TYPE "public"."audit_action" RENAME VALUE 'item.created' TO 'event.created_youtube';
ALTER TYPE "public"."audit_action" RENAME VALUE 'item.deleted' TO 'event.deleted_youtube';
```
Pro: forward-only, no UPDATE rows. Con: invents new audit verbs; the audit log shows `event.created_youtube` for legacy rows and `event.created` for new rows — confusing for the user.

**Path B — UPDATE rows then DROP enum values** (cleaner audit trail):
```sql
-- Step 1: change rows to use the new vocabulary BEFORE dropping enum values.
UPDATE "audit_log" SET "action" = 'event.created'  WHERE "action" = 'item.created';
UPDATE "audit_log" SET "action" = 'event.deleted' WHERE "action" = 'item.deleted';
-- Step 2: drop the now-unused item.* enum values.
-- Postgres has NO native DROP VALUE — workaround is the textcast-then-recreate dance.
-- BUT: with zero production data and only a handful of dev rows, this is acceptable.
ALTER TABLE "audit_log" ALTER COLUMN "action" SET DATA TYPE text;
DROP TYPE "audit_action";
CREATE TYPE "audit_action" AS ENUM (
  'session.signin', 'session.signout', 'session.signout_all', 'user.signup',
  'key.add', 'key.rotate', 'key.remove',
  'game.created', 'game.deleted', 'game.restored',
  'event.created', 'event.edited', 'event.deleted', 'event.attached_to_game', 'event.dismissed_from_inbox',
  'source.added', 'source.removed', 'source.toggled_auto_import',
  'theme.changed'
);
ALTER TABLE "audit_log" ALTER COLUMN "action" SET DATA TYPE "audit_action" USING "action"::"audit_action";
```

**Recommendation: Path B.** AGENTS.md invariant 4 says audit log is INSERT-only and tenant-relative — it does NOT prohibit **schema-level** UPDATE during a forward-only migration; it prohibits the application code from issuing UPDATE/DELETE against the table at runtime. Migrations are exempt by definition (every migration is a one-time DDL/DML operation under a different code path). Plan precedent: the project's existing `0001_phase02_schema.sql` already issued `ALTER COLUMN ... SET DATA TYPE` on `audit_log.action`. The Phase 2 STATE.md decision "drizzle-kit 0.31 emitted ALTER COLUMN ... TYPE audit_action USING action::audit_action automatically — RESEARCH.md Open Question 2 worst case (manual hand-edit) did not materialize" confirms the textcast pattern works on this codebase's pinned drizzle-kit version.

The companion migration is therefore Path B (UPDATE rows + textcast recreate), hand-written, with the new vocabulary ALSO including the new Phase 2.1 verbs:
- `source.added`, `source.removed`, `source.toggled_auto_import` (replacing channel.*)
- `event.attached_to_game`, `event.dismissed_from_inbox` (new workflow verbs from §3.4)

## 5. SOURCES-01 Forward-Compat Strategy

**Confidence: MEDIUM** (UX call; both paths defensible, but one materially safer for the user).

### 5.1 The question

SOURCES-01 says the schema accepts five `kind` values but only `youtube_channel` is functional in Phase 2.1. What does the UI do when the user picks `reddit_account` from the kind dropdown?

### 5.2 Recommendation: SHOW non-functional kinds as DISABLED chips with "Coming in Phase X" copy

**Rationale:**
1. **Hiding kinds creates surprise.** Phase 2 UAT showed (PROJECT.md "Model evolved during Phase 2 UAT") that the user thinks in terms of platforms, not "what we've built". A dropdown that hides 4 of 5 kinds will be re-discovered ("can I add Reddit?") and look like a regression when Phase 3 lands them.
2. **Disabled chips telegraph the roadmap.** "Reddit (Coming in Phase 3)" is honest, sets expectations, and is internally consistent with the Phase 2 dashboard's "Phase 4 will add charts" pattern.
3. **It's cheap.** One Paraglide key per kind for the "coming-soon-phase" string; `<button disabled>` is one prop.

### 5.3 API behaviour: 422 on non-functional kind in POST /api/sources

When the user (or, more importantly, an api-test or rogue client) submits `POST /api/sources {kind: 'reddit_account'}`:

- **Locked: return 422** with code `'kind_not_yet_functional'` and metadata `{kind, available_phase: 'Phase 3'}`.
- **NOT 200**: a passive 200 would silently persist the row; Phase 3's polling worker would later try to poll a Reddit account with no Reddit OAuth secret configured (KEYS-02 deferred), which would dump an `auth_error` into the events stream confusingly.
- **NOT 501 (Not Implemented)**: 501 implies "the server doesn't understand the request"; we understand it perfectly, we just choose not to enable it yet. 422 (Unprocessable Entity) is semantically correct.

## 6. INBOX-01 Dismissal Schema

**Confidence: HIGH** (jsonb namespace pattern is unambiguous; index decision is a clear "no" at indie scale).

### 6.1 Locked metadata jsonb shape

```typescript
// events.metadata jsonb shape (per-kind union)
type EventMetadata = {
  // Inbox dismissal — NAMESPACED to avoid colliding with per-kind keys.
  inbox?: {
    dismissed: boolean;       // INBOX-01 — sets to true when user clicks "Mark not game-related"
    dismissed_at?: string;    // ISO8601 audit aid
  };
  // Per-kind stats (Phase 3 worker writes these).
  youtube?: {
    view_count: number;
    like_count: number;
    comment_count: number;
    last_polled_at: string;
  };
  reddit?: { upvotes: number; num_comments: number; last_polled_at: string; };
  // ... etc per kind
};
```

**Namespacing decision:** `inbox.dismissed` (nested) NOT `inbox_dismissed` (flat). Rationale:
1. The flat key was the orchestrator's strawman (`metadata->>'inbox_dismissed'`). It works but doesn't compose — adding `inbox_dismissed_at` later means a SECOND flat key, which couples them by convention only.
2. Nesting under `inbox` gives a clear audit-friendly home for related fields (`dismissed_at`, future `dismissed_reason`, etc.) and makes per-feature jsonb additions explicit.
3. Postgres `metadata->'inbox'->>'dismissed'` is one extra `->` operator; query plans on jsonb are unaffected at indie scale.

### 6.2 Query patterns

```sql
-- /feed inbox view (attached=false): exclude dismissed.
SELECT * FROM events
WHERE user_id = $1
  AND game_id IS NULL
  AND deleted_at IS NULL
  AND COALESCE(metadata->'inbox'->>'dismissed', 'false') = 'false'
ORDER BY occurred_at DESC, id DESC
LIMIT 51;

-- "Show all inbox including dismissed" (advanced filter — Phase 3+, NOT Phase 2.1):
SELECT * FROM events WHERE user_id = $1 AND game_id IS NULL AND deleted_at IS NULL
ORDER BY occurred_at DESC, id DESC;
```

### 6.3 Index decision

**Locked: NO jsonb index in Phase 2.1.**

Reasoning:
- The `attached=false` view's filter `metadata->'inbox'->>'dismissed' = 'false'` is the only jsonb-keyed predicate in Phase 2.1.
- At indie scale (single user, expected ≤10k events lifetime per ROADMAP success criteria), the `events_user_occurred_at_idx` (added in §1) lets the planner do an index scan + filter — Postgres evaluates the COALESCE on `metadata->'inbox'->>'dismissed'` only on the rows already filtered to `user_id = $1 AND game_id IS NULL`. That's <10k rows worst-case; jsonb scan is negligible.
- A GIN index on `metadata` would add storage cost and write amplification on every event update for a query path that's already <10ms.

**Phase 6+ revisit trigger:** if `attached=false` queries exceed 50ms p95 in Loki metrics, add `CREATE INDEX events_inbox_partial_idx ON events (user_id, occurred_at DESC) WHERE game_id IS NULL AND COALESCE(metadata->'inbox'->>'dismissed', 'false') = 'false'`. File as Phase 6 deferred item.

## 9. Migration File Layout

**Confidence: HIGH** (split is forced by §4 — drizzle-kit can't auto-emit ALTER TYPE RENAME VALUE).

### 9.1 Locked: 2-file split

```
drizzle/
├── 0000_flimsy_ezekiel_stane.sql          (Phase 1 — unchanged)
├── 0001_phase02_schema.sql                (Phase 2 — unchanged)
├── 0002_phase02_1_data_sources.sql        (NEW — auto-generated)
└── 0003_phase02_1_audit_enum_rename.sql   (NEW — hand-written companion)
```

### 9.2 0002 (auto-generated via `pnpm db:generate`)

Contains:
- `ALTER TABLE youtube_channels RENAME TO data_sources` (interactive prompt: Y)
- `ALTER TABLE data_sources RENAME COLUMN is_own TO is_owned_by_me` (interactive prompt: Y)
- `CREATE TYPE source_kind AS ENUM (...)` + `ALTER TABLE data_sources ADD COLUMN kind / auto_import / metadata / deleted_at`
- `DROP TABLE game_youtube_channels`
- `DROP TABLE tracked_youtube_videos`
- `ALTER TYPE event_kind ADD VALUE 'youtube_video'` + `ADD VALUE 'reddit_post'`
- `ALTER TABLE events ADD COLUMN author_is_me / source_id / last_polled_at / last_poll_status`
- `ALTER TABLE events ALTER COLUMN game_id DROP NOT NULL`
- `CREATE INDEX events_user_occurred_at_idx`, `events_user_source_idx`, `data_sources_user_kind_idx`, `data_sources_user_deleted_at_idx`
- Renames of indexes that drizzle-kit auto-emits (`youtube_channels_user_id_idx` → `data_sources_user_id_idx`, etc.)

### 9.3 0003 (hand-written, runs immediately after 0002 under same advisory lock)

Contains the audit_action enum rename per §4.3 Path B:
- `UPDATE audit_log SET action = 'event.created' WHERE action = 'item.created';` (and the deletion variant)
- `ALTER TABLE audit_log ALTER COLUMN action SET DATA TYPE text;`
- `DROP TYPE audit_action;`
- `CREATE TYPE audit_action AS ENUM (...new vocabulary...);`
- `ALTER TABLE audit_log ALTER COLUMN action SET DATA TYPE audit_action USING action::audit_action;`
- (No re-creation of the (user_id, action, created_at) composite index — `ALTER COLUMN ... SET DATA TYPE` preserves indexes by Postgres semantics.)

### 9.4 Why split (vs single-file)

Postgres DDL is transactional; both files COULD be one file. But:
1. **drizzle-kit's `generate` would never produce the §4 enum rename** — keeping them separate makes the hand-written content reviewable in isolation.
2. **Future audits**: when a maintainer reads `git log drizzle/`, "0002 = data sources realignment" and "0003 = audit enum vocabulary" is more legible than "0002 = both".
3. **Roll-forward retry safety**: if 0002 succeeds and 0003 fails on the first deploy, the operator can fix 0003 and re-run; if both were one file, the table-rename would be reverted and re-applied (safe, but slower).

The Phase 1 `runMigrations()` advisory lock applies per-file but each file is atomic; both files will commit serially in the same migration run.

### 9.5 Drizzle journal hygiene

The hand-written `0003_phase02_1_audit_enum_rename.sql` MUST be added to `drizzle/meta/_journal.json` (the journal that `runMigrations()` consumes). drizzle-kit doesn't auto-add hand-written files — the planner's task author MUST update the journal (project precedent: STATE.md "drizzle migrator requires _journal.json on disk to discover migrations; without it, runMigrations() is a silent no-op"). Verify with `pnpm db:check` (commutativity check from drizzle-kit 0.31).

## Common Pitfalls

(Captured here to satisfy the standard RESEARCH.md format; the load-bearing ones are surfaced inline above.)

### Pitfall 1: drizzle-kit silent enum-rename failure
**What goes wrong:** Renaming an enum value in `pgEnum(...)` declaration; drizzle-kit emits broken USING-cast SQL that fails on existing rows.
**Why it happens:** [drizzle-orm#5340](https://github.com/drizzle-team/drizzle-orm/issues/5340) — drizzle-kit doesn't yet emit `ALTER TYPE ... RENAME VALUE`.
**How to avoid:** Hand-write a companion migration using Postgres-native `ALTER TYPE ... RENAME VALUE` syntax, OR if collisions force it, use UPDATE + textcast-recreate (§4.3 Path B). Add to `drizzle/meta/_journal.json` so `runMigrations()` discovers it.
**Warning signs:** `pnpm db:migrate` fails with "invalid input value for enum" on dev DB containing UAT rows.

### Pitfall 2: Forgetting to update `eslint-plugin-tenant-scope/no-unfiltered-tenant-query.js` TENANT_TABLES
**What goes wrong:** Renamed `youtubeChannels` → `dataSources` in code; ESLint rule still references the old name; `dataSources` queries pass lint without userId filter; cross-tenant leak ships.
**Why it happens:** TENANT_TABLES is a hand-maintained Set (Phase 2 STATE.md decision); rename refactors don't auto-propagate.
**How to avoid:** In the SAME plan that adds the schema rename, update `eslint-plugin-tenant-scope/no-unfiltered-tenant-query.js`: REMOVE `'youtubeChannels'` and `'gameYoutubeChannels'` and `'trackedYoutubeVideos'`; ADD `'dataSources'`. RuleTester suite (already exists from Plan 02-02) extends with at least one positive case for `dataSources` and one negative (allowlist) case to ensure the rule matches the new table.
**Warning signs:** RuleTester suite passes but a query against `dataSources` without userId filter doesn't trigger an error during lint of `src/lib/server/services/data-sources.ts`.

### Pitfall 3: Forgetting to update `MUST_BE_PROTECTED` allowlist on route rename
**What goes wrong:** Anonymous-401 sweep allowlist still references `/api/items-youtube` and `/api/youtube-channels`; new routes /api/sources / /api/events/:id/attach / /api/events/:id/dismiss-inbox don't appear; sweep is vacuous-pass on the new routes.
**Why it happens:** Phase 2 Plan 02-08 STATE.md noted "MUST_BE_PROTECTED gains the new routes" — the same discipline must repeat in Phase 2.1.
**How to avoid:** Treat the allowlist update as a checklist item in the route-refactor plan: REMOVE old entries; ADD new entries; the sweep test asserts `protectedRoutes.length >= N`.
**Warning signs:** Anonymous-401 test passes but a curl with no cookie against `/api/sources` returns 200/empty.

### Pitfall 4: Per-event game attach without re-asserting cross-tenant 404
**What goes wrong:** PATCH /api/events/:id/attach takes a `gameId` payload; if the service doesn't `await getGameById(userId, gameId)` before setting `events.game_id = gameId`, a forged payload could attach the caller's event to another tenant's game (well, the FK would catch it as a 23503, but the response would be 500 not 404).
**Why it happens:** Mistaking "tenant scope on events" for "tenant scope on the attach target".
**How to avoid:** `attachEventToGame(userId, eventId, gameId, ipAddress)` MUST call `await getGameById(userId, gameId)` BEFORE `db.update(events).set({gameId})...`; the existing items-youtube precedent has this exactly right and is the model.
**Warning signs:** Cross-tenant attach test returns 500 not 404.

### Pitfall 5: Auto-decrement count-of-attached-events when soft-deleting a data source
**What goes wrong:** SOURCES-02 adds soft-delete to data_sources; existing events with `source_id = X` keep that source_id even after X is soft-deleted. /feed source-filter for the deleted source still shows those events with a "ghost" source name.
**Why it happens:** `source_id` is a FK with `ON DELETE SET NULL`, but soft-delete doesn't trigger `ON DELETE`.
**How to avoid:** /feed source filter chip dropdown excludes `data_sources WHERE deleted_at IS NOT NULL` (so users can't pick a ghost source). The events themselves keep `source_id` pointing at the soft-deleted row — that's correct (audit trail of "this event came from that source, even though the source is now retired"). Optional Phase 6 polish: a "purge orphans" path during account deletion.
**Warning signs:** /feed shows events whose source_id points at a soft-deleted source; UI either says "(deleted source)" or just hides the source chip.

### Pitfall 6: Not updating `EventKind` const VALID_EVENT_KINDS in services/events.ts
**What goes wrong:** Services-level VALID_EVENT_KINDS const (Phase 2 events.ts) doesn't include `youtube_video` / `reddit_post`; a runtime call passing `kind='youtube_video'` throws `validation_failed` 422 even though Postgres would have accepted it.
**Why it happens:** Two sources of truth: the pgEnum declaration (schema) and the const in services/events.ts (Phase 2 STATE.md decision: "defense-in-depth — both layers reject the same set").
**How to avoid:** In the same plan that extends `event_kind` enum, update VALID_EVENT_KINDS to mirror it. The mirror is intentional (defense-in-depth), but it MUST be kept in sync — file a Plan 02-1-X "single-source-of-truth helper that exports both" if drift becomes a recurring pattern.
**Warning signs:** Schema accepts `'youtube_video'` but services-level validation rejects it.
