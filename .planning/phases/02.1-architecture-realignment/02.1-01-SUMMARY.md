## Decisions Made

- **events.game_id ON DELETE SET NULL (not CASCADE).** Phase 2 had `events.game_id NOT NULL ... ON DELETE CASCADE`; Phase 2.1 makes it nullable AND switches to SET NULL. This is the same change in both directions: deleting a game now returns its events to the inbox (game_id NULL) instead of cascading them away. Aligns with the inbox-first IA — events outlive their games.
- **events.source_id ON DELETE SET NULL.** Soft-removing a data source must not destroy events that came from it (the events themselves are user-authored content, not source-derived data). The events stay in the timeline with `source_id IS NULL` (looking like manual paste). Trade-off: forensic loss of "which source originally imported this" — acceptable; the audit log already has the import event.
- **Partial unique index on data_sources(user_id, handle_url) WHERE deleted_at IS NULL.** Without the partial clause, soft-deleting a source would block the user from re-adding the same handle later. With it, the user can resurrect a removed source by re-adding it.
- **Partial unique index on events(user_id, kind, source_id, external_id).** Both `source_id` and `external_id` are nullable; a non-partial unique would either reject manual-paste rows (no source_id) or allow many rows with NULL (Postgres NULL semantics) — neither matches the dedup intent. The partial clause restricts dedup enforcement to the auto-import path where both fields are present.
- **Single new baseline collapse over RESEARCH §4 two-file dance.** Per CONTEXT D-03 / DV-1, the Phase 2.1 destructive baseline replaces RESEARCH §4's Path B `ALTER TYPE ... RENAME VALUE` workaround. Saves ~200 lines of hand-written SQL + the entire migration mechanic dance. Cost: AGENTS.md forward-only exception scoped to Phase 2.1 ONLY; Phase 3+ resumes strict forward-only.
