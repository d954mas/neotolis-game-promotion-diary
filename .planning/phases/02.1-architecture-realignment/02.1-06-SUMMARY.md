## Decisions Made

- **Per-game events list lives on gamesRoutes (not eventsRoutes).** `GET /api/games/:gameId/events` is registered alongside `gamesRoutes`'s `/games/:id/restore` so Hono's path-matching handles the parameterized form unambiguously: `/games/:id` (bare-id form) is registered first, `/games/:gameId/events` (parameterized form with sub-path) is registered second. This mirrors the Phase 2 `gameListingsRoutes` mount discipline.
- **HTTP-boundary tests are NEW describe blocks alongside existing service-layer ones.** The plan's `<action>` Step C said "Flip the remaining `it.skip(...)` entries" but the actual placeholder files (data-sources / feed / inbox / events-attach) had ALREADY been flipped to live SERVICE-level tests in Plans 02.1-04 and 02.1-05 — those plans named the flipped tests `Plan 02.1-04:` / `Plan 02.1-05:` because they owned the service-layer work. So the grep contract (`it.skip("Plan 02.1-06:` returns 0) was already satisfied. This plan ADDED separate `Plan 02.1-06: HTTP boundary` describe blocks that exercise the same contracts through `app.request(...)` — service-layer + HTTP-layer coverage in one file each.
- **zod v4 record schema syntax.** The plan's `<action>` skeleton showed `z.record(z.unknown())` (v3 single-arg form). v4 requires `z.record(z.string(), z.unknown())` — explicit key + value schemas. Both routes/sources.ts and routes/events.ts use the v4 form. Caught by tsc on the first build attempt.
- **DELETE /api/sources/:id returns 200 + soft-deleted DTO** (not 204). The service contract (Plan 04 — `softDeleteSource` returns the soft-deleted row) lets the UI render the retention badge from the response without a follow-up GET.
- **The cross-tenant matrix's user B owns its own game (`gameB`) so the Pitfall 4 probe is realistic.** A B-cookie request that PATCHes `/api/events/${userA_event}/attach` with `body.gameId = gameB.id` exercises the case where the gameId IS legitimate for the caller — but the EVENT belongs to A, so the UPDATE matches no rows and NotFoundError fires. This is the exact scenario where a naive implementation (without `assertGameOwnedByUser` or with the FK check FIRST) would surface a 500.
- **services/games.ts cascade fix is in scope.** The plan didn't list `services/games.ts` in `<files>`, but the route mount chain transitively imports it. The cascade UPDATEs against `gameYoutubeChannels` + `trackedYoutubeVideos` (retired in Plan 02.1-01) made `createApp()` throw during integration test runs. Rule 3 (blocking) — fix it inline. Confined to the cascade lines; no other functions touched.
