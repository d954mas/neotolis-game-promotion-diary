# Phase 2.1 — UI Design Contract

> Visual and interaction contract for the architecture-realignment phase. Generated by gsd-ui-researcher; consumed by gsd-ui-checker, gsd-planner, gsd-executor.
>
> **Scope.** Phase 2.1 retires the per-platform `youtube_channels` + `tracked_youtube_videos` mental model and ships the unified `data_sources` + `events` model with three primary views (`/feed`, `/sources`, `/games/[id]` rebuilt). Plus bundled UI polish (sessions list, /keys/steam empty-state copy, event delete ConfirmDialog, AppHeader avatar+email, theme toggle moved to /settings).
>
> **Stack reality (UNCHANGED from Phase 2).** SvelteKit 2.58 + Svelte 5 (runes) + Paraglide JS 2 + native CSS. No Tailwind, no shadcn, no React, no component library. Indie-budget constraint (PROJECT.md) and locked-stack constraint (RESEARCH.md §"Standard Stack") forbid adding any new top-level UI dependency in Phase 2.1.
>
> **Inheritance.** This spec INHERITS the entire Phase 2 design contract verbatim — spacing scale, typography, color tokens, Layout & Responsive Contract (incl. 360px hard floor), Form interaction contract, Theme toggle interaction (with one delta — see "Theme toggle relocation" below), Accessibility floor, ConfirmDialog speedbump pattern. The sections below specify ONLY what 2.1 changes or adds.
>
> **Aesthetic stance (unchanged).** Tool-feel, terminal-leaning, dense information OK, calm > flashy, privacy-only with zero growth surfaces. Empty-state copy is practical (real example URLs).

## Color

**UNCHANGED tokens, three new usage rules.**

The 60/30/10 split, all CSS custom properties, and dark/light/system theme handling stay identical to Phase 2 (`src/app.css` not edited). New components consume the existing tokens.

| Role | New 2.1 surfaces |
|------|------------------|
| Accent (10%) | Primary CTAs of new pages: "+ Add data source" on /sources, "Save" on RenameInline + AddSteamListingForm; active-state of new Nav items (Feed default landing → underline border in accent) |
| Destructive | event delete ConfirmDialog confirm button (NEW polish item — Phase 2 shipped delete without a confirm; 2.1 fixes it); "Soft-delete data source" confirm button on /sources |
| Info (`--color-info`) | The `<PollingBadge>` Phase-3 placeholder text uses `--color-text-muted`, NOT info; reserved for "you registered a source but auto-import is OFF" inline note on /sources detail |
| Text-muted | InboxBadge background (`--color-border`) + text (`--color-text-muted`); SourceKindIcon stroke; FilterChip text when not active; PollingBadge placeholder |

**Accent NOT for (UNCHANGED + new exclusions):**
- AttachToGamePicker open-state highlight — uses `--color-border` darkening, not accent.
- InboxBadge — uses muted, not accent. The point is "this awaits action," not "this is featured."
- The `is_owned_by_me=true` badge on SourceRow — uses muted text + bordered chip; we don't want "owned" to look like a status award.
- FilterChip when active — uses surface fill + 1px border in `--color-text`; saturated accent is too loud at this density.

**Why the restraint.** /feed will frequently render 50 rows of mixed kind/source/game chips. If every chip used accent the page becomes a Christmas tree and the user can't scan. Accent stays reserved for primary CTAs and the navigation-active underline; everything else is bordered surface + muted text.

## Layout & Responsive Contract — DELTA from Phase 2

The 360px hard floor (UX-02), no-horizontal-scroll guarantee, page max-width 1024px, and the breakpoint set (`--bp-mobile` 360 / `--bp-tablet` 768 / `--bp-desktop` 1024) all carry forward. Phase 2.1 adds three patterns the Phase 2 spec didn't have.

### `/feed` filter row — chips → sheet pattern

> Manual UAT verification per VALIDATION.md: "/feed at 360px (authenticated)" — sign in via local dev server, resize to 360×640, verify chips collapse to "Filters (N)" sheet button.

- **`min-width: 600px`:** all 7 active-filter chips render inline above the row list, horizontally scrollable if more than ~5 are active. A "Clear all filters" link sits at the right end of the chip strip.
- **`max-width: 599px`:** chips collapse into a single `<button>Filters (N)</button>` (where N = active filter count). Tapping the button opens `<FiltersSheet>` — a `<dialog>`-based modal anchored to the bottom of the viewport on mobile, full-screen on `<360px`. Sheet has every filter as labelled controls (`<select>` for source/kind/game; `<input type="date">` for from/to; toggles for attached + author_is_me); "Apply filters" button submits → `goto(/feed?...)` with the new params; "Clear all" resets all and submits.
- **Filter changes always reset cursor (RESEARCH §3.4 (b)).** A new filter combination is a new query, not a paginated continuation.
- **Single-filter dismiss:** desktop chips have an `×` (44×44 hit area) that removes that one filter and re-navigates.

### `/feed` row interaction contract

The single most-rendered surface in 2.1 — a /feed page renders 50 of these per page. The contract optimizes for scan-ability + tappability at 360px.

1. **Row layout (mobile, 360px):** vertical stack inside a `<li>` —
   - Line 1: `<KindIcon>` + `<time>{occurredAt}</time>` + `<InboxBadge>` (if applicable)
   - Line 2: title (body 16, font-weight regular; truncate with ellipsis at row width)
   - Line 3: source chip (if source_id) + game chip (if game_id) + author_is_me badge — each wraps if more than viewport
   - Line 4: actions row — `<AttachToGamePicker>` (left) + `<button>Open</button>` (link to event detail stub) + edit + delete (icon-only, 44×44)
2. **Row layout (≥768px tablet):** horizontal flexbox — KindIcon · timestamp · title (flex-grow 1, ellipsis) · source/game chips · author badge · actions cluster on right. Single line height.
3. **No row click → page navigation** (Phase 2 precedent — explicit affordances only). The "Open" button on each row links to `/events/[id]` (Phase 4 lands the detail page; Phase 2.1 ships a stub that says "Detail page lands in Phase 4").
4. **AttachToGamePicker affordance:** closed state = compact button (`<icon>` + label "Attach to game" or game name). Open state = anchored dropdown listing the user's games (max 10 shown, scrollable beyond). Last two options separated by a divider: "Move to inbox" (clears game_id) and "Mark not game-related" (calls dismiss-inbox endpoint). Esc closes.
5. **Delete affordance always shows ConfirmDialog** (UI polish fix — Phase 2 shipped one-click delete without confirm; 2.1 fixes it across all event-delete paths via `<FeedRow>`).
6. **Same component, same rules on `/games/[id]`** — Phase 2.1 reuses `<FeedRow>` on the curated per-game view; the only difference is rows are grouped under `<MonthHeader>` instead of paginated by cursor.

### `/sources` page layout

- Header: `<h1>Data sources</h1>` + `<button>+ Add data source</button>` (top-right desktop; sticky-bottom on mobile per Phase 2 pattern).
- Body: list of `<SourceRow>`. Empty state via `<EmptyState>`.
- Add flow: clicking "+ Add data source" navigates to `/sources/new` (no inline form — gives the kind picker enough room and matches the Phase 2 `/keys/steam` add-via-form pattern). The form has: kind picker (functional kinds enabled, 4 disabled with phase tooltips per RESEARCH §5.2), `display_name` text input, `external_url` text input (e.g. handle URL), `is_owned_by_me` toggle (default ON), `auto_import` toggle (default ON if `is_owned_by_me=true`, else OFF), submit + cancel.
- Source-row layout (mobile 360px): vertical stack — Line 1: SourceKindIcon + display_name + Mine/Tracking badge. Line 2: handle URL (monospace 14px). Line 3: PollingBadge + auto_import toggle. Line 4: edit + remove icons (44×44).
- Source-row layout (≥768px): horizontal — SourceKindIcon · display_name (flex-grow) · handle URL (muted) · Mine/Tracking · PollingBadge · auto_import toggle · edit/remove cluster.

### `/games/[id]` rebuild — closes the P0 functional gap

The Phase 2 page existed but lacked rename UI and add-Steam-listing UI. The 2.1 rebuild fixes this and pivots from the Phase 2 panel-stack to an event-curated layout.

- Header panel:
  - Game title rendered via `<RenameInline>` (click-to-edit, Save/Cancel/Esc/Enter contract above).
  - Tag chips + release-date badge below the title.
  - Notes textarea (read-mode + edit-mode toggle from Phase 2, unchanged).
- Store-listings panel:
  - List of existing `game_steam_listings` rows.
  - **`<AddSteamListingForm>`** at the bottom — paste field for Steam URL → fetches appdetails → creates listing. Failure modes use `<InlineError>` (existing Phase 2 component).
- Curated events panel:
  - Heading: "Events (N)".
  - Rows grouped by `<MonthHeader>` (e.g. "April 2026 — 7 events"); within a group, rows are `<FeedRow>` (same component used on /feed).
  - Empty state when no events: `<EmptyState>` with `m.empty_feed_filtered_heading()` + a custom body explaining "Attach events from the /feed page to surface them here."
  - **No paste box on this page in 2.1.** Manual paste lives on /feed (where the user is already chronologically oriented). Optional: a small "+ Paste URL on this game" affordance that links to `/feed?game={id}&attach_pending=true` and pre-fills the game on attach. Planner discretion.
- YouTube channels panel — **REMOVED**. The `game_youtube_channels` M:N table is dropped (RESEARCH §1.1). The new model attaches events to games, not sources to games.
- Tracked-items panel — **REMOVED**. `tracked_youtube_videos` is dropped; events of `kind=youtube_video` attached to this game render in the curated events panel.

### `<AppHeader>` — UI polish (UserChip + theme toggle removal)

- Layout (mobile 360px): app title (link to `/feed`) on the left + UserChip (avatar 28px circular, email hidden if `<480px`) on the right. UserChip is a `<button>` that opens a small menu (`<dialog>` again or simple absolutely-positioned panel) with: email (read-only), "Sign out", "Sign out from all devices". Each menu action is 44px tall.
- Layout (≥768px): app title left + UserChip right with email visible (label 14px, semibold).
- `<ThemeToggle>` is REMOVED from AppHeader. It now lives only on `/settings` (UI polish: the Phase 2 UAT P1 surfaced the toggle was clutter on every page; settings is the right home).

### Theme toggle relocation

- Phase 2 theme-toggle interaction contract is preserved (three states: light/dark/system; cookie + DB write on click; SSR-flash mitigation via `transformPageChunk`; `prefers-color-scheme` honored only when `data-theme="system"`).
- The component renders ONLY on `/settings` now. The `m.theme_toggle_aria_label` and `m.theme_label_*` keys carry forward.
- `/settings` Theme section (existing per Phase 2) gets a sub-blurb (`m.settings_theme_blurb()` above) and the toggle inline.

### Default-route swap (`/` → `/feed`)

Per RESEARCH §3.6: authenticated users hitting `/` are redirected to `/feed`. No UI change on `/` itself — the redirect happens in `+page.server.ts`. Anonymous users still land on `/login`. Visual contract: nothing rendered on `/` for authenticated users (instant redirect), Phase 1 placeholder renders for anonymous.

## Accessibility Floor — DELTA

The Phase 2 floor (semantic HTML, focus-visible 2px accent outline, AAA color contrast, programmatic form labels, `<time datetime>`, `<dialog>` modal focus trap) carries forward unchanged. New 2.1 surfaces add:

- **`<FeedRow>` is a list item.** The /feed body is `<ul>`; each row is `<li>`. The "Open", edit, delete controls are `<button>`s; the "Attach to game" trigger is a `<button>` that opens a `<dialog>` (mobile) or absolutely-positioned panel with `role="menu"` (desktop). Keyboard: Tab cycles row controls; Enter/Space activates; Esc closes any open picker.
- **`<FilterChips>` are `<button>`s** with `aria-pressed="true"` when active. Dismiss × is a separate `<button>` inside the chip with `aria-label="Remove {filter} filter"`.
- **`<FiltersSheet>` is a `<dialog>` with `aria-labelledby` pointing at the sheet heading.** The first focusable control inside the sheet receives focus on open. Esc closes.
- **`<MonthHeader>` is `<h2>`** (page heading is `<h1>` "Game / {title}"; month groups within are `<h2>`). Screen readers announce the group.
- **`<SourceKindIcon>` and `<KindIcon>`** carry `aria-hidden="true"` (decorative); the kind name is conveyed in adjacent text via Paraglide labels.
- **`<UserChip>`** is a `<button>` with `aria-label="Account menu — {email}"`; the avatar `<img>` has empty `alt=""` (decorative, label is on the button).
- **`<InboxBadge>` and `<PollingBadge>`** are `<span>`s with `role="status"` so screen readers announce state changes when a row's badge updates.

Color-contrast computation for new chip/badge surfaces (light theme):
- InboxBadge: `--color-text-muted` (#666666) on `--color-border` (#e5e5e5) = 4.55:1 (AA) — meets the floor.
- PollingBadge placeholder: same combination, same ratio.
- FilterChip active: `--color-text` (#1a1a1a) on `--color-surface` (#ffffff) = 16.1:1 (AAA).
- Mine/Tracking badge: `--color-text-muted` on `--color-bg` = 4.92:1 (AA).

Dark theme equivalents all clear AA per the same computation.

## Flags (non-blocking observations for the planner)

- **FLAG: 24 new Paraglide keys.** The Phase 2 keyset was 80 keys (per Phase 2 plan 02-09 STATE entry). Phase 2.1 adds approximately 50 new keys (5 audit-action labels + 9 event-kind labels + 5 source-kind labels + 5 disabled-kind phase labels + 8 filter labels + 9 page CTAs + 8 empty-state copy keys + 5 sessions-section keys + …). The keyset snapshot test from Phase 2 (D-41) MUST be expanded in Wave 0 to assert the new alphabetical list. Recount during plan-phase; the budget is "the test asserts what the executor ships."

- **FLAG: AttachToGamePicker open state on mobile.** The picker dropdown could be inline (anchored beneath the trigger) OR a bottom-sheet `<dialog>`. The contract above leaves this to planner discretion. Recommendation: inline anchored dropdown is fine when the user has ≤10 games (the common case); bottom-sheet only if the user has >10 games (rare in 2.1 — first-time users will have 0–3 games). Defer the bottom-sheet variant to Phase 6 polish unless executor finds the inline dropdown clipping at 360px.

- **FLAG: `/feed` row "Open" button.** Phase 2.1 ships the FeedRow's Open button but has no /events/[id] detail page — Phase 4 owns VIZ-01. The Open button in 2.1 navigates to `/events/[id]` which renders a stub: `<EmptyState>` heading "Detail page lands in Phase 4" + body explaining LayerChart + a "Back to feed" link. Honest, sets expectations, avoids a 404. Alternatively, the Open button could be hidden in 2.1 and shipped in Phase 4 — planner picks. Recommendation: ship the stub (the affordance is part of the row's information architecture, not contingent on chart UI).

- **FLAG: PollingBadge default for `kind=youtube_video` events with `source_id=NULL` (manual paste).** Three states are possible: (a) the badge says "Manual entry — no polling" (`m.polling_badge_manual()`); (b) the badge is hidden entirely (no source = no polling = no status to show); (c) the badge says "Phase 3 — paste polling stub" (suggests a Phase 3 enhancement that doesn't exist yet). Recommendation: **(a)** — explicit muted text "Manual entry — no polling" is honest and matches the "calm > flashy" stance. Hidden creates surprise; option (c) overpromises.

- **FLAG: Filter chip overflow at 360px.** With 7 axes × at most 7 active simultaneously, the chip strip can overflow even at desktop. The `min-width: 600px` chip-inline rule keeps mobile clean (sheet handles all 7), but at 600–800px, 5 active chips can wrap to a second line. The contract permits this — `flex-wrap: wrap` on the chip container is fine. Don't fight it.

- **FLAG: `/sources/new` "kind picker" disabled-chip UX.** RESEARCH §5.2 locks "show disabled with 'Coming in Phase X' copy" — this contract honors it. The disabled chips are `<button disabled>` with `aria-disabled="true"` and `tabindex="-1"`; tooltip on hover (or focus on keyboard) shows the phase status copy. Don't render them as a `<select>` option that throws on submit — that's worse UX. Render as visible chips so the user sees the roadmap.

- **FLAG: AppHeader email visibility breakpoint.** The contract says email hides at `<480px`. The standard breakpoint set has no 480px — it's a one-off. Consider adding `--bp-mobile-tight: 480px` to the token set in `src/app.css` for this single use, or hard-code 480px in the AppHeader's media query. Recommendation: hard-code; this is the only place 480px is used and the token would be premature.
