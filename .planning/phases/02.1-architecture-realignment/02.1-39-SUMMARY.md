# Tech tracking
tech-stack:

  patterns:
    - "Single .sticky-chrome wrapper pattern — when multiple top-bar elements (AppHeader + Nav) need to act as one cohesive sticky block, wrap them in a single position:sticky parent in +layout.svelte rather than stacking independent sticky children. Eliminates subpixel-rounding gap-vs-overlap trade-offs that plagued polish #1-#4. Children render in document flow inside the wrapper; the wrapper is the only sticky element. CSS variable --chrome-height (ResizeObserver-measured) carries the wrapper's runtime height to descendants (e.g., FeedDateGroupHeader's top: calc(--chrome-height + ...))."
    - "RecoveryDialog modal as the universal soft-delete recovery surface — when a feature has soft-deleted entities AND infinite-scroll lists, render recovery in a native <dialog> + showModal() rather than a bottom-of-page <details> block. Decouples recovery UX from scroll position by construction. PageHeader exposes the trigger via `onOpenRecovery: () => void`; consumers wire `entityType=\"event\"|\"game\"|\"source\"|\"store\"`. Pattern absorbs growth: future entity types extend RecoveryDialog's entityType union, not a new component."
    - "Instagram-style sticky date-group header — when a chronological list groups by date, the group header sticks at the BOTTOM of the chrome+PageHeader stack (not behind it). FeedDateGroupHeader: position:sticky; top: calc(--chrome-height + --page-header-height - --sticky-overlap). Each group's header rotates as the next group scrolls into view (browser native sticky behavior). User-quoted reference: 'Так в гугл таблицах или инстаграмме сделоано'."
    - "Modal-based add/edit pattern over inline forms — when a form lives inside a per-entity detail page (e.g., AddStoreDialog inside /games/[gameId]), render it in a native <dialog> triggered from a section-header CTA rather than always-visible inline fields. Cleaner section header, less vertical space, modal scope matches user intent ('I want to add a thing'). AddStoreDialog + GameEditDialog both follow the ConfirmDialog precedent — `.dialog[open]` CSS scope prevents closed-dialog flash."

## Issues Encountered

- **Subpixel rounding fragility across DPR / zoom levels.** Four polish iterations (#3-#6) chased the math before the architectural collapse in #5/#7 made the issue moot. Lesson: when a sticky stack has more than one boundary, prefer collapsing to one sticky parent over fighting subpixel arithmetic.
- **Path A recovery anchor incompatible with infinite scroll.** Plan-time §5.8 chose Path A as the cheap fix; UAT immediately surfaced the incompatibility. Lesson: "cheap fix" choices made at plan time without considering scroll-loader interactions are high-risk.
- **Schema deferred at plan time can become required at UAT time.** games.description was deferred to Phase 6 in the §5.3 plan-time framing; round-6 walkthrough re-surfaced it as essential. Forward-only migration is cheap (`drizzle-kit migrate` is idempotent) so adding it inline rather than waiting was the right call.
