## Phase Boundary

Retire Phase 2's per-platform mental model (`youtube_channels` + `tracked_youtube_videos` + per-game M:N attach + dashboard-at-`/`) and ship the unified `data_sources` + `events` model with the three-view IA (`/feed` primary, `/sources` registry, `/games/[id]` rebuild). Schema migration is destructive but cheap — zero production data, zero self-host deployments — so the executor wipes the dev DB and applies a single new baseline migration with the final 2.1 schema.

Phase 3 lands the polling worker that actually fills the inbox, the YouTube/Reddit API key UIs (KEYS-01, KEYS-02), the Reddit URL ingest (INGEST-01), and the per-event stats history (event_stats_snapshots) that Phase 4 charts read from. Phase 2.1's `DataSourceAdapter` interface is the seam — Phase 2.1 ships the type + a `youtube_channel` STUB that throws "implemented in Phase 3"; Phase 3 fills in `pollContent` / `pollStats` for both source-attached events AND manual-paste pollable events (D-06).

## Implementation Decisions

### Phase Shape & Waves

- **D-01 — Single Phase 2.1 (no 2.1a/2.1b split).** ~10–12 plans organized in 5 waves (Wave 0 schema+scaffolding; Wave 1 services; Wave 2 routes; Wave 3 UI pages; Wave 4 polish; Wave 5 smoke+verify). One atomic package, no intermediate "half-migrated" master state. Mirrors Phase 2 precedent (11 plans / 4 waves) and AGENTS.md "atomic PRs, one clear goal" rule. The user explicitly rejected the 2.1a/2.1b split because the intermediate state ("new routes without UI") would leave the dev product partially broken between phases.
- **D-02 — Wave 0 = migration + scaffolding; Wave 1+ rewrites on top.** Wave 0 lands the single new baseline SQL migration (D-03), updates `eslint-plugin-tenant-scope/no-unfiltered-tenant-query.js` TENANT_TABLES (`youtubeChannels` / `gameYoutubeChannels` / `trackedYoutubeVideos` removed; `dataSources` added), lands 5 new test placeholder files per VALIDATION.md (data-sources.test.ts, feed.test.ts, inbox.test.ts, events-attach.test.ts, browser/feed-360.test.ts) as `it.skip` placeholders, and re-affirms (no edit needed) AGENTS.md "Privacy & multi-tenancy" invariants. Wave 1+ deletes old services / routes and builds new ones on the post-migration schema. Between waves CI may fail locally, but each wave-merge to the phase branch produces a green CI; the phase merges to master only after Wave 5.

### Migration Mechanics — DEVIATION from RESEARCH §4

- **D-03 — Single new baseline migration (DEVIATION from RESEARCH §4 two-file split).** RESEARCH §4 prescribed `0002_phase02_1_data_sources.sql` (auto-generated forward-only) + `0003_phase02_1_audit_enum_rename.sql` (hand-written companion using `ALTER TYPE ... RENAME VALUE` Path B to work around drizzle-orm#5340). The user has decided that — because Phase 2 is dev-only and zero self-host deployments exist — the cleaner path is to **delete `drizzle/0000_flimsy_ezekiel_stane.sql` + `drizzle/0001_phase02_schema.sql` and generate one new baseline** (`drizzle/0000_phase02_1_baseline.sql`) containing the final 2.1 schema (data_sources with `kind`/`is_owned_by_me`/`auto_import`/`metadata`/`deleted_at`; events with `author_is_me`/`source_id`/`last_polled_at`/`last_poll_status`/nullable `game_id`; audit_action enum with the final 2.1 vocabulary; no youtube_channels / tracked_youtube_videos / game_youtube_channels). This eliminates the `ALTER TYPE RENAME VALUE` workaround entirely (drizzle-kit 0.31 cleanly emits an enum CREATE without collisions). RESEARCH §4 dance is **NOT applicable** to Phase 2.1.
- **D-04 — Dev DB wiped during Phase 2.1 first-load.** Executor's plan README explicitly instructs the developer (and CI) to run `pnpm db:drop && pnpm db:migrate` before the first 2.1 service starts. UAT data from Phase 2 (game cards, registered YouTube channels, tracked items, events) is intentionally lost — that data was scratchpad for Phase 2 surface validation, not production state. CI's Postgres service container starts empty per build, so `db:drop` is a no-op there. The smoke test asserts the unified flow from a clean DB.
- **D-03 implication for AGENTS.md "migrations forward-only" rule.** AGENTS.md says "Migrations forward-only, run at boot under advisory lock". D-03 collapses 2 prior migrations into a new baseline — this is **NOT forward-only** by the strict definition. The user accepts this deviation because (a) the codebase is pre-launch, (b) zero self-host deployments exist that would replay the old 0000+0001 in sequence, and (c) the alternative (RESEARCH §4 hand-written companion) carries higher review cost for no reachable user benefit. Phase 3+ migrations resume the forward-only contract from the new baseline (`0001_phase03_*`, `0002_phase04_*`, etc.).

### PollingBadge UX

- **D-05 — Single PollingBadge contract for pollable kinds, regardless of source_id.** PollingBadge text (`m.polling_badge_phase3_placeholder()` — copy: `Phase 3 will start polling`) renders on every event with `kind ∈ {youtube_video, reddit_post}` AND `last_polled_at IS NULL`, regardless of whether `source_id` is set or NULL. PollingBadge is **hidden** for non-pollable kinds (`conference`, `talk`, `press`, `other`, `twitter_post`, `telegram_post`, `discord_drop` — none of these have a poll adapter in v1; Twitter is permanently out per PROJECT.md). Phase 3 replaces the placeholder with real "Hot/Warm/Cold/Stale" surface (POLL-05). **Overrides UI-SPEC FLAG** that proposed three options for manual-paste events; the user clarified that manually-pasted YouTube videos MUST get stats history too, so the badge behaviour is unified.
- **D-06 — Phase 3 polling worker scope: pollStats covers all pollable events independent of source_id.** RESEARCH §2.1 already locked `pollStats(event: EventRow, source: DataSourceRow | null)` with nullable source. D-05 makes this load-bearing for the user contract: a manually pasted blogger video (`source_id=NULL, author_is_me=false`) gets the same stats history treatment as a source-attached own video (`source_id=X, author_is_me=true`). Phase 3 worker scope must enumerate `events WHERE kind IN ('youtube_video', 'reddit_post') AND deleted_at IS NULL` and call `pollStats` regardless of `source_id`. Implementation detail (per-event `videos.list?id=…` for orphan events vs batched per-source playlist polling for source-attached events) is Phase 3's call; Phase 2.1 ships only the schema + interface + STUB.

### `/events/[id]` Detail Page Stub

- **D-07 — Ship Phase 4 stub at `/events/[id]` in Phase 2.1.** `<FeedRow>` "Open" button navigates to `/events/[id]`; the page renders an `<EmptyState>` with copy "Detail page lands in Phase 4 — graphs and per-event metric history arrive with VIZ-01" + a "Back to feed" link. This keeps the FeedRow's IA complete (the affordance is part of the row's information architecture per UI-SPEC FLAG) and avoids dead-end 404s. Phase 4 replaces the stub with the LayerChart-driven detail page sourced from `event_stats_snapshots` (Phase 3 table). Two new Paraglide keys for the stub: `m.events_detail_phase4_heading()` / `m.events_detail_phase4_body()`.

### AttachToGamePicker + "+ New Event"

- **D-08 — AttachToGamePicker mobile open-state: inline anchored dropdown.** No bottom-sheet `<dialog>` in Phase 2.1. The dropdown anchors below the FeedRow's "Attach to game" trigger, scrollable internally beyond ~10 items. Bottom-sheet variant (better for users with many games) is **filed as Phase 6 polish item**, triggered by UAT discovery of clipping at 360px on users with >10 games (rare in early MVP). Saves UI code surface in 2.1.
- **D-09 — "+ New event" free-form creation: separate route `/events/new` (NOT inline dialog on /feed).** Consistent with `/sources/new` pattern (UI-SPEC). Both `/sources/new` and `/events/new` are full-page forms. Reduces "two patterns for creating a resource" decision in the codebase. `/feed` header gets a "+ New event" button (Paraglide key `m.feed_cta_new_event()` — copy: `+ New event`) that navigates to `/events/new`. Form has 9-kind picker (with non-pollable kinds enabled by default — pollable kinds via paste-box per Phase 2 INGEST flow), title, occurredAt, optional URL, optional notes, optional game attach. Submit → POST `/api/events` → navigate to `/feed`. Cancel → navigate back to `/feed` via `m.common_cancel()`.

### Smoke Test Scope

- **D-11 — Phase 2.1 smoke covers the unified register→paste→see→attach→see flow only; dismiss-inbox deferred to Phase 3 smoke.** CI smoke (`tests/smoke/self-host.sh`) extension: register YouTube data_source via POST `/api/sources` → paste YouTube URL via the existing INGEST flow (now creating events with `source_id=NULL`, `kind=youtube_video`) → assert event appears in GET `/api/events` (the /feed query) → PATCH `/api/events/:id/attach` with `{gameId}` → assert event appears in GET `/api/games/:gameId/events` (the /games/[id] curated view). The PATCH `/api/events/:id/dismiss-inbox` happy path is covered by `tests/integration/inbox.test.ts` only; smoke for it lands in Phase 3 alongside the polling worker (the worker is what populates the inbox naturally, making the smoke flow honest end-to-end).

### Onboarding & Cross-Resource UX

- **D-12 — No YouTube API key warning on `/sources` in Phase 2.1.** PollingBadge text `Phase 3 will start polling` is the only signal in 2.1 that polling isn't live yet. Adding an inline-info note "Polling needs a YouTube API key in Phase 3 — add it on /keys when ready" duplicates Phase 3 onboarding (KEYS-01 ships its own onboarding flow). Saves one Paraglide key + one inline UI element + cognitive load in 2.1. The user can register a source today and the app does not pretend the polling will start before Phase 3.
- **D-13 — `/feed` filter persistence: URL params only (no cookie/localStorage).** Each `/feed` visit is a clean slate (default filters: none → all events sorted DESC). Bookmarking a filtered URL is the persistence mechanism. Saves cookie name + reconciliation logic + self-host parity surface (one fewer cookie path to document). If UAT surfaces "I want my last filters remembered" as a real pain point, Phase 6 polish adds a cookie-backed lastFilters with URL-wins reconciliation (mirrors the Phase 2 D-40 cookie+DB theme pattern).
- **D-14 — 0 games + source registration: allowed (no onboarding gate).** A user can register their first `data_source` before creating their first `game`. AttachToGamePicker on `/feed` rows shows an inline link "No games yet — + Add a game" (Paraglide key `m.feed_attach_no_games_inline()` — copy: `No games yet — + Add a game`) when the user's `games` list is empty. Source and game are independent top-level concepts (PROJECT.md "три независимых top-level lists"); forcing creation order would contradict the data model. The /feed empty-state copy already explains the inbox/attach pattern (UI-SPEC `m.empty_feed_body()`).
