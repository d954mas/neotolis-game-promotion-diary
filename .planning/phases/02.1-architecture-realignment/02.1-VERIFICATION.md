## Verification context

This is NOT the standard `gsd-verifier` automated VERIFICATION.md. It was
authored by the orchestrator after a manual UAT walk-through against the
Plan 02.1-10 checkpoint, where the user surfaced gaps that automated tests
did not catch. The format mirrors verifier output so `/gsd:plan-phase 02.1
--gaps` can consume it directly.

The gap-closure plan should treat each gap as a phase task. Gaps are tagged
P0 (blocks Phase 2.1 sign-off — broken contracts, runtime crashes, design
discrepancies that the user explicitly rejected) or P1 (UX polish surfaced
during UAT that the user wants in scope before sign-off). One Phase 3+
backlog item is recorded separately.

A small set of inline hot patches was applied during the UAT session and
landed in commits `fc8354b` (dev catch-all for `/api/*`) and `f52f1b4`
(Phase 2.1 hot patches: Nav reshuffle, /feed CTA, FilterChips visibility,
date-range inclusive-bounds + min/max cross-binding, /events/new datetime
→ date, dark-theme date-picker icon, /settings Activity-log link). Those
fixes unblocked the UAT itself; gap-closure does NOT need to redo them.

## Gaps — P0 (blocks sign-off)

### Gap 1 — `/audit` page crashes at runtime

**Symptom:** `TypeError: m.audit_action_item_created is not a function`
when the audit page renders. The dropdown in `<ActionFilter>` and the
row label in `<AuditRow>` both call Paraglide keys
`m.audit_action_item_created` / `m.audit_action_item_deleted`, which were
removed in Plan 02.1-03 (because Plan 02.1-01 dropped the `item.*` audit
actions from the enum).

**Why automated tests missed it:** no integration test exercises the
`/audit` page render. Plan 02.1-09's test extensions covered empty-states,
theme, and events — not audit. Plan 02.1-06's anonymous-401 + cross-tenant
sweeps cover the API, not page render.

**Fix scope:**
- `src/lib/components/ActionFilter.svelte`: drop the two `item.*` rows; add
  the five new Phase 2.1 audit actions (`source.added`, `source.removed`,
  `source.toggled_auto_import`, `event.attached_to_game`,
  `event.dismissed_from_inbox`) per UI-SPEC §"`/audit` (UNCHANGED
  structurally; copy delta)".
- `src/lib/components/AuditRow.svelte`: same — replace the `item.*` cases
  with the new audit-action mapping; render the new Paraglide keys.
- `tests/integration/audit.test.ts`: add a render-level assertion that
  catches a runtime key reference to a missing Paraglide key (a smoke
  render of `<ActionFilter>` and `<AuditRow>` for every value of
  `AUDIT_ACTIONS` is enough).

**Requirements traceability:** AUDIT-01 (Phase 2 carry-forward).

### Gap 2 — Soft-delete event recovery UI is missing

**Symptom:** UI-SPEC §"Destructive Confirmations" promises the user
"Delete this event? You can restore it within 60 days." in the event-delete
ConfirmDialog. The user can soft-delete an event, but there is nowhere to
view or restore it. The promise is currently a lie.

**Reference contract:** `/sources` already ships the soft-deleted-section
toggle pattern ("Show N deleted sources" expand block with `<RetentionBadge>`
and Restore button — Plan 02.1-08). The same pattern should apply to events.

**Fix scope:**
- Decide where deleted events live: most natural place is `/feed` itself
  (a "Show N deleted events (within 60 days)" expand toggle below the
  CursorPager). Alternative: a `/feed/deleted` sub-route. Plan-time call.
- `src/lib/server/services/events.ts`: extend `listFeedPage` (or add a
  parallel `listDeletedEvents`) that returns rows with `deletedAt IS NOT
  NULL AND deletedAt > now() - retention_days * interval`. Same
  `userId`-first tenant scope discipline.
- `src/lib/server/services/events.ts`: add `restoreEvent(userId, eventId)`
  symmetric to the data-sources restore — clears `deletedAt`, audit-logs
  `event.restored` (new audit-action verb — also needs the Paraglide
  label key).
- HTTP: `PATCH /api/events/:id/restore` (analogous to
  `PATCH /api/sources/:id/restore` from Plan 02.1-06).
- UI: extend `<FeedRow>` (or a new `<DeletedFeedRow>`) with a Restore button
  + `<RetentionBadge>` showing remaining days.
- Tests: extend `tests/integration/events.test.ts` with restore + retention
  assertions; add tenant-scope probe for cross-tenant restore (404 not 500).

**Requirements traceability:** RETN-01 (retention contract — events were
already covered by the schema's `deleted_at` column from Plan 02.1-01,
just not surfaced in UI / API).

### Gap 3 — Feed renders as text rows; user wants media cards

**Symptom:** UI-SPEC line 386 specified "label-14 typography, lighter
weight, muted text" for FeedRow — text-dense rows. After landing, the user
explicitly said "хочется видеть фид как карточки" with media previews:
YouTube events should show the video thumbnail.

**Why this is in scope for Phase 2.1, not Phase 6:** The user is rejecting
the implemented design after seeing it. UI-SPEC is approved, but design
intent is more important than spec adherence when the user is the design
authority. Without this redesign the /feed surface — the new primary
landing page for the entire product — feels wrong.

**Fix scope:**
- New `<FeedCard>` component (or major rewrite of `<FeedRow>`) with:
  - Thumbnail/preview slot (YouTube: `https://img.youtube.com/vi/{id}/
    mqdefault.jpg` — public CDN, no API key, deterministic from URL).
  - Title row (existing).
  - Source / game / inbox / polling chips row (existing semantics, new
    layout — chips below the thumbnail+title block).
  - Actions row (Attach / Edit / Delete — bottom-right of card).
- Fallback for events without thumbnail (conference, talk, press, other,
  generic post): text-only card with kind icon as visual anchor.
- Mobile: full-width stacked cards. Desktop: same width as today, just
  different internal layout.
- Update `tests/browser/feed-360.test.ts` to assert the card structure.
- Update `<KindIcon>` consumers — see Gap 6.

**Requirements traceability:** FEED-01 (the page itself). UI-SPEC delta
required.

### Gap 4 — Multi-select filters (kind / source / game)

**Symptom:** User requested checkbox-style multi-select for the kind,
source, and game filters. Current single-select dropdowns force the user
to pick one value per axis at a time — they can't see "all youtube_video
events from sources A, B, C" or "all events for games X and Y" in one view.

**Fix scope:**
- Backend: extend `FeedFilters` interface to accept arrays:
  `source: string | string[] | undefined`, same for `kind` and `game`.
- SQL: switch from `eq(...)` to `inArray(...)` when the value is an array;
  preserve `eq(...)` for the single-value backwards-compatible path.
- URL encoding: `?source=a&source=b` (URLSearchParams `getAll`) or
  `?source=a,b` (comma split). Pick one and document.
- `+page.server.ts` + `routes/events.ts`: parse the new shape; update tests.
- `<FiltersSheet>`: replace the three `<select>` inputs with checkbox
  lists. For source / game with many entries, add a small typeahead /
  filter input.
- `<FilterChips>`: render one chip per active value (so source=[A,B,C]
  produces three dismissable chips); dismiss removes a single value.
- Tests: extend `tests/integration/feed.test.ts` to cover multi-select
  semantics (empty array = no filter; one value = same as current; many
  values = OR semantics within axis, AND across axes).

**Requirements traceability:** FEED-01 (filter contract).

## Gaps — P1 (UX polish surfaced during UAT)

### Gap 5 — Compact / relative date format on FeedCard

**Symptom:** "Дата много места занимает". Full ISO datetime is loud.

**Fix scope:**
- Helper `formatFeedDate(occurredAt)` in `src/lib/util/`:
  - Today → `"Today, 14:30"`
  - Yesterday → `"Yesterday"`
  - This year → `"Apr 25"`
  - Older → `"Apr 25, 2025"`
- Use in `<FeedCard>` and `<FeedRow>`-likes.

### Gap 6 — KindIcon doesn't communicate "YouTube" / per-platform identity

**Symptom:** Geometric non-brand KindIcon (UI-SPEC §"Inline-SVG icon
additions") chosen to avoid brand-mark licensing concerns. User can't
distinguish a YouTube video from a Reddit post at a glance.

**Fix scope:** Two paths, planner picks one:
- **(a)** Sublty restyle the existing geometric icons with platform-tinted
  borders (red ring around play-button for YouTube, orange for Reddit) —
  gives signal without licensing exposure.
- **(b)** Add a small text label next to the icon: "YouTube" / "Reddit" /
  etc., using the existing `m.event_kind_label_*()` Paraglide keys. No
  brand-mark; the kind taxonomy speaks platform names.
- Likely converges with Gap 3 (when there's a thumbnail, kind is implicit;
  the label is the fallback for non-thumbnail kinds).

### Gap 7 — Today / Yesterday quick buttons on `/events/new`

**Symptom:** User has to type or click through the calendar for "today" /
"yesterday" — the two most common picks.

**Fix scope:** Two `<button type="button">` chips above the date input
that set `occurredAt` to today / yesterday. Trivial.

### Gap 8 — UI feedback when date is auto-derived from a YouTube URL

**Symptom:** When user pastes a YouTube URL, the resulting event takes the
video's publish-date (existing `createEventFromPaste` behaviour, Plan
02.1-05). User finds this confusing — they couldn't tell where the date
came from. Either:

**Fix scope:**
- **(a)** Always default to "today" on the form; `occurredAt` only auto-
  populates if the user explicitly checks "Use publish date".
- **(b)** Auto-populate from URL but show an inline badge "Date from
  YouTube publish-date — you can change it" near the field.

(b) preserves the convenience; (a) is simpler. Plan-time call.

### Gap 9 — Default `/feed` filter = last 30 days

**Symptom:** No default filter → all events ever returned. User wants
"last 30 days" as the default view, dismissable for a full archive.

**Fix scope:**
- `src/routes/feed/+page.server.ts`: when neither `from` nor `to` is in
  the query string, set `from = today - 30 days`, `to = today`.
- Render the resulting filter as an active chip "Last 30 days" so the
  user can dismiss it (which navigates to `/feed?all=1` or similar
  marker that opts out of the default).
- Tests: assert the default filter is applied on first visit and can be
  cleared.

### Gap 10 — Date-range as primary control, not in FiltersSheet

**Symptom:** User says "часто видел что дату делают отдельно от фильтров".
The standard analytics / feed pattern is a prominent date-range selector
above the list with quick presets (Today / Last 7d / Last 30d / Custom),
with the rest of the filters in a sheet behind a "Filters" button.

**Fix scope:**
- New `<DateRangeControl>` component above the FilterChips strip on
  `/feed` (and reusable on other date-filtered surfaces in future).
  Quick-preset chip row + "Custom" opens the date-range fields.
- `<FiltersSheet>` loses its `from` / `to` fields (they move to the
  primary control).
- Migrates Gap 9's "Last 30 days" default into a preset selection — when
  the user hasn't customized, the "Last 30 days" preset is the active one.

### Gap 11 — Auto-test for `/audit` page render (regression guard)

**Symptom:** Gap 1 was a runtime crash that no automated test caught.

**Fix scope:** As part of fixing Gap 1, add a render-level integration
test: bootstrap the SvelteKit page, assert `<ActionFilter>` renders one
option per `AUDIT_ACTIONS` value, and assert `<AuditRow>` renders without
throwing for every action value with a synthetic row. This is a Nyquist
sample — tests every audit-action enum value, so future enum drift is
caught.

### Gap 12 — Add new event kind `post`

**Symptom:** User wants a generic `post` kind for platforms not yet
mapped (Mastodon, LinkedIn, Bluesky, Threads). Currently the only escape
hatch is `other`, which loses platform-agnostic post categorization.

**Fix scope:**
- Schema: add `post` to the `event_kind` enum (forward-only migration —
  this is the first migration AFTER the Phase 2.1 baseline collapse).
- `src/lib/server/db/schema/events.ts`: extend the TS enum.
- `src/lib/server/services/events.ts`: add `post` to `VALID_EVENT_KINDS`.
- `messages/en.json`: add `m.event_kind_label_post()` Paraglide key.
- `<KindIcon>`: add a generic post SVG (paper / document silhouette).
- `<ActionFilter>`: no change (audit actions are not per-kind).
- Tests: extend the kind-enum mirror assertion and the kind-picker test.

## Phase 3+ backlog

### Multi-game per event

User wants one event (e.g., a publisher's YouTube video) to attach to
multiple games at once. Currently `events.game_id` is a single nullable
FK. This requires either:
- A `game_ids[]` array column on events (Postgres-friendly, simpler
  migration), or
- A junction table `event_games(event_id, game_id, attached_at,
  attached_by)` (canonical N:M).

**Decision:** This is a structural change that ripples into Phase 3's
adapter polling (a YouTube video imported via auto-import resolves to
ZERO games initially — N games could be auto-attached based on title
matching, source ownership, etc.). Bundle it into the Phase 3 design.
