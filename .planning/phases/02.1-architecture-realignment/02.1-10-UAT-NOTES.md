### 8.4-backlog — Multiple credentials per service-type (label + default-key)
- Status: backlog, Phase 3+ scope (affects credentials schema + hub UI)
- User quote: "Да и может быть несколько кредентиалов на тип?"
- Use cases:
  - **Steam**: developer with multiple studios — separate API keys per studio account.
  - **YouTube**: multiple API project keys to spread quota across, or per-game promo channel.
  - **Reddit**: multiple bot accounts for different community engagement strategies.
  - **itch.io**: one account per game / per studio.
- Schema implication for `api_keys_*` tables (currently one row per user):
  - Drop the implicit one-per-user constraint (if any); allow N rows per user per service.
  - Add `label TEXT NOT NULL` (display name like "Studio A live", "Test environment", "Beta channel API"); UNIQUE on `(user_id, service, label)`.
  - Add `is_default BOOLEAN NOT NULL DEFAULT false`; partial unique index `(user_id) WHERE is_default = true` per service-table to enforce at-most-one default per user per service.
  - DataSourceAdapter / poll workers pick the `is_default` key for that user+service (or accept an explicit `api_key_id` argument).
- UI implication for `/settings/credentials` hub (8.2-redesign):
  - Each service section becomes a **list** of credentials, not a single row.
  - Each credential row: label · status · last-used · is-default badge · Manage / Set-as-default / Remove.
  - "Add another" button per service.
- Migration: not breaking — existing single-key rows get `label = "default"`, `is_default = true`.
- Bundles with the broader credentials-hub work in 8.2-redesign and Phase 3+ integrations.

### 6.3-bug — Sign-out-this-session does not reactively refresh the sessions list
- Status: UX bug, candidate Phase 6 polish backlog
- Severity: minor — sign-out itself works (server-side session is destroyed), only the local UI state lags.
- Observed on `/settings` two-browser scenario (UAT 2026-04-29):
  1. Two browsers signed in → both sessions appear in the list correctly.
  2. Click "Sign out this session" on the OTHER (non-current) row → server destroys that session.

- User quote: "Работает. Но я выбрал sign out, сессия завершилась, но ui не обновился пока я не обновил окно"

- Suggested fix:
  - Use SvelteKit's `enhance` action with `applyAction` + `invalidateAll()` on success in `src/routes/settings/+page.svelte` (the sign-out form).
  - OR have the action return a `redirect(303, '/settings')` to force fresh load.
- Files: `src/routes/settings/+page.svelte`, possibly `src/lib/components/SessionsList.svelte`.

### 6.1-redesign — Inbox triage UX: per-card actions + new "standalone" state with dim styling
- Status: UX redesign (extends 1.5-redesign cluster) + new data state
- Severity: not a Phase 2.1 functional blocker (current attach-from-edit works), but user explicitly wants quick triage on the feed without round-tripping to edit.

**Two changes proposed:**

1. **Inbox cards get inline actions** (an exception to the otherwise read-only feed rule established in 1.5-redesign):
   - "Attach to game" — opens picker → sets `game_id` → row exits inbox.
   - "Not related to any game" / "Standalone" — marks the event as triaged-but-unattached → row exits inbox AND becomes visually dimmed in `/feed`.
   - **Reason for the exception**: triaging the inbox is a frequent batch operation; forcing the user to open `/events/[id]/edit` for each one is too slow. Attach/edit/delete OUTSIDE the inbox stays in edit (as in 1.5).

2. **New event state: "standalone-marked" / "not related to any game"**
   - Currently "inbox" is defined as `game_id IS NULL` (per RESEARCH §6 and the listFeedPage filter on line 834-841 of `events.ts`).
   - Proposed: split that into two states:
     - **Inbox** = `game_id IS NULL AND NOT standalone_marked` → user hasn't triaged yet.
     - **Standalone** = `game_id IS NULL AND standalone_marked = true` → user explicitly said "this is not tied to any game" (e.g. an industry conference talk, a press piece about studios in general, a personal milestone).
   - Storage: a boolean flag on `events` (could be a new column `is_standalone` OR `metadata.triage.standalone = true` like the existing `metadata.inbox.dismissed` pattern). Probably the metadata path is cleaner since it follows precedent.
   - Visual: standalone events render **dimmed** (lower opacity / grey-out) in `/feed` so they don't distract from game-tied events.
   - Filter: `/feed` Show fieldset gains a "Standalone" option (alongside Inbox / Specific games / Any).

- User quote: "Так и в фиде. Я вот хочу для inbox значений. Чтобы была кнопка добавить к игре, или чтобы можно было пометить, ка не связанное с игрой.(такие не связанные с игрой, нужно как-то затемнять, чтобы они не мешали)"

### 2.5-backlog — Source-relink + identity by external_id (channel_id) + post-ingest "register channel?" prompt + per-source aggregate stats
- Status: backlog cluster (NOT a Phase 2.1 blocker — affects scope of Phase 3+)
- Severity: design enhancement, currently safe-but-incomplete: orphan events without source links remain orphan forever; events created BEFORE source registration are not retroactively re-linked.

**Backlog 2.5.a — Use platform-stable `external_id` as the identity key, not just `handleUrl`**
- Current: `enrichFromUrl` matches `oEmbed.author_url` against `data_sources.handleUrl` (URL string equality).
- Problem: YouTube handles can change (`@oldname` → `@newname` rebrand), custom URLs vary, URL trailing-slash and case differences cause false negatives.
- Proposed: each `data_source` carries an `external_id` column populated at registration time:
  - `youtube_channel` → YouTube channel ID like `UCxxxxxxxx` (resolve via YouTube Data API or oEmbed when registering).
  - `reddit_account` → username string.
  - `twitter_account` → numeric user ID (handle changes don't break it).
  - `telegram_channel` → channel username or numeric ID.
  - `discord_server` → guild ID.
- Match flow becomes: `enrichFromUrl` extracts `external_id` from the URL (parse + maybe API lookup); compare against `data_sources.external_id` for exact match. Falls back to handleUrl match for legacy rows.
- Migration: add `data_sources.external_id` column nullable; backfill on next polling cycle for existing rows.
- User quote: "По сути мы тогда для ютуба храним какой-то id канала, и может связать с источником?"

**Backlog 2.5.b — Re-link orphan events on source registration (and after)**
- Trigger: when a user registers a new `data_source`, scan all their existing events with `source_id IS NULL` and `metadata.author_url` (or `metadata.external_id` once 2.5.a lands) matching the new source → SET source_id, author_is_me retroactively.
- Also offer one-time "re-link existing orphans" button on `/sources/[id]` for sources registered before this feature lands.
- Audit log: `event.source_relinked` action per re-linked event.

**Backlog 2.5.c — Post-ingest "register channel?" prompt**
- After `createEventFromPaste` returns with `source_id = NULL`, the response includes a hint with the parsed `author_url` + `author_name`.
- UI shows a non-blocking toast or inline banner: "Detected channel @smoke. Register as a data source?" → one click creates the `data_source` AND triggers backlog 2.5.b re-link for the just-created event.
- Closes the loop so users don't accumulate orphan events.

**Backlog 2.5.d — Per-source aggregate stats (channel-level time series) — Phase 3+ companion to event_stats_snapshots**
- Per AGENTS.md / Architecture: Phase 3+ ships `event_stats_snapshots` (per-event time series). The user asked whether we can track aggregate channel-level metrics separately: "и можем еще отдльено трекать всю статистику канала для всех?"
- Proposed: parallel table `data_source_stats_snapshots` (per-source time series) capturing channel-level metrics:
  - `youtube_channel` → subscribers count, total views, video count.
  - `reddit_account` → karma, post count, follower count.
  - `twitter_account` → followers, following, tweet count.
- Polling adapter populates this on the same schedule as event polling (or a slower cadence — channel-level metrics don't change by the second).
- Charts: `/sources/[id]` gets a sparkline panel showing channel growth alongside the event list.
- Privacy: tenant-scoped exactly like `event_stats_snapshots` (each user has their own snapshot rows for their registered sources).

**Combined impact**: this cluster turns Sources from a passive registry into a useful tracking surface. None of it blocks 02.1 ship.

### 5.5 — failed (P1) `body:has(dialog[open]) { overflow: hidden }` does NOT block mouse-wheel scroll

Plan 02.1-34's CSS lock only applies `overflow: hidden` to `body`, not to `html` / `:root`. In modern browsers the document scroll-root is `<html>`, so wheel events keep scrolling `<html>` even when `<body>` is overflow-locked.

**User quote:** "скролл работает под окном, колесико мышки"

- `src/app.css:194-196` — `body:has(dialog[open]) { overflow: hidden; }` only.

**Suggested fix (round-6 gap closure):** mirror the rule onto `:root` (or `html`). One-line patch:

```css
:root:has(dialog[open]),
body:has(dialog[open]) {
  overflow: hidden;
}
```

Both selectors needed because some browsers (Chrome ≤ 122) only honor `:has()` on the originating element if it has the corresponding rule at root. Test in Chrome (Win), Firefox, and Safari at 360px AND desktop wheel-scroll. **Round-6 polish follow-up #2 (2026-04-30, post-runtime-measurement):** after the ResizeObserver landed in commit `273904d`, the user re-walked item 1 and reported a NEW symptom — verbatim: "Теперь есть gap в 1-2 пикселя между заголовком и табами, через него я вижу контент скролла" ("Now there's a 1-2 pixel gap between the header and the tabs, through which I see the scrolling content").

Root cause: `HTMLElement.offsetHeight` is specified to return an **integer** number of CSS pixels (floor-rounded). AppHeader's actual rendered height is fractional in real browsers — the `--space-md` padding (16px × 2) plus the 28px avatar plus the brand text line-box plus a 1px border that scales with DPR can land on a non-integer total like 77.5px or 77.33px depending on device-pixel-ratio (zoom level, HiDPI display, browser-zoom). `offsetHeight` then floors to 77; Nav stickies at `top: 77px` while AppHeader's real bottom edge sits at 77.5px → a 0.5px gap that visually reads as 1-2px because subpixel anti-aliasing widens the visible seam. Same fix applied symmetrically to `--nav-height` so PageHeader.sticky's `top: calc(var(--app-header-height) + var(--nav-height))` also rounds up at every level of the stack.

**Fix (this commit):**
- `src/routes/+layout.svelte` — drop `Math.ceil(...)`; write the raw fractional `getBoundingClientRect().height` value directly into `--app-header-height` / `--nav-height`. - `src/app.css` — declare a new `--sticky-overlap: 1px` design token under `:root`. This is the belt-and-suspenders safety net for any future browser-engine drift or HiDPI edge case.
- `src/lib/components/Nav.svelte` — `.nav` `top: calc(var(--app-header-height, 72px) - var(--sticky-overlap, 1px))`. Nav overlaps AppHeader by 1 invisible pixel.
- `src/lib/components/PageHeader.svelte` — `.page-header.sticky` `top: calc(var(--app-header-height, 72px) + var(--nav-height, 44px) - 2 * var(--sticky-overlap, 1px))`. PageHeader is two tiers deep, so it subtracts 2 × overlap (one per tier above it).

**Why overlap is invisible:** AppHeader, Nav, and PageHeader.sticky all paint with matching backgrounds (`--color-surface` or `--color-bg`) plus the existing `border-bottom` that already sits at the boundary. The 1px overlap column paints the same surface color twice — pixel-identical to no overlap. The border-bottom of the upper tier and the top edge of the lower tier coincide on the same pixel row, which is what the user expects visually anyway.

### 5.12 — failed (P1) `createEvent` and `attachEventToGames` perform multi-step writes without a transaction

Both service functions perform several DB writes that must succeed-or-fail together but do not wrap them in `db.transaction(...)`:

A. **`createEvent` (events.ts:332-…)** — INSERT into `events` then (Plan 02.1-28 path) loop INSERT into `event_games` then `writeAudit`. If the events INSERT succeeds but a junction INSERT 23505s (duplicate composite PK on a retry path) or the audit write fails (FK to user, network blip), we keep the orphan parent row + partial junction. The "validate-first; INSERT only after pass" rule (CLAUDE.md anti-pattern list, item 5) is honored for ownership/dedup, but doesn't help on multi-row write atomicity.

B. **`attachEventToGames` (events.ts:1081-1198)** — DELETE existing junction rows (loop), INSERT new rows (loop), UPDATE parent `updatedAt`, write audit (loop, one per diff row). Five logical phases, none in a transaction. A FK violation on INSERT (race with another tab dropping the game between `assertGameOwnedByUser` and the INSERT) would leave the junction half-deleted.

**Suggested fix (round-6):** wrap the multi-step body of each function in `db.transaction(async (tx) => { ... })`. Drizzle exposes `tx` with the same surface; replace `db.insert(...)` / `db.delete(...)` / `db.update(...)` with `tx.*` inside the closure. The audit writes can stay outside the transaction since `writeAudit` is INSERT-only and tenant-relative — a missing audit row on a successful business write is a forensics-acceptable failure mode (per AGENTS.md privacy floor item 4 — audit failure must not block business path).

Decision: audit write OUTSIDE transaction (current behavior preserved). Junction + parent inside.

**Closure plan target:** round-6 plan — `db.transaction()` wrap on both service functions + race-condition integration test (concurrent attach/detach).
