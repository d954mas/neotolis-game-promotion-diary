# Phase 02.1 Plan 15: Multi-select feed filters + date-range primary control + last-30-days default Summary

Backend FeedFilters interface extended to accept `string | string[]` (and `EventKind | EventKind[]`) for source / kind / game; HTTP layer reads repeated query params via Hono's `c.req.queries()`; /feed page-server parses arrays via `URLSearchParams.getAll()` and applies a last-30-days default when no from/to/all=1 is supplied; new `<DateRangeControl>` becomes the primary date control above the chip strip; `<FilterChips>` emits ONE chip per VALUE for the multi-select axes; `<FiltersSheet>` replaces select dropdowns with checkbox lists (with typeahead for source/game).

## Closes (from VERIFICATION.md)

- **Gap 4 (P0):** multi-select filters per axis — closed.
- **Gap 9 (P1):** last-30-days default on first /feed visit — closed.
- **Gap 10 (P1):** date-range as primary control above filter chips — closed.

## Backend signature delta

```typescript
// BEFORE (Plan 02.1-05)
export interface FeedFilters {
  source?: string;
  kind?: EventKind;
  game?: string;
  // ...
}

// AFTER (Plan 02.1-15)
export interface FeedFilters {
  source?: string | string[];
  kind?: EventKind | EventKind[];
  game?: string | string[];
  // ...
}
```

`pushAxis<T>(parts, column, value)` helper added next to `assertValidKind`. It branches on the value shape:

- `undefined` → no clause appended.
- `[]` (empty array) → no clause appended (= "nothing selected" matches all).
- `[x]` (1-elem array) → `eq(column, x)` (preserves the single-string query plan).
- `[x, y, ...]` → `inArray(column, [x, y, ...])`.
- bare scalar → `eq(column, value)` (back-compat for existing callers).

`listFeedPage` swaps the three single-axis if-blocks for three `pushAxis(...)` calls. The userId WHERE clause stays lexically present in `.where(...)` so the `tenant-scope/no-unfiltered-tenant-query` ESLint rule keeps recognizing it; pushAxis builds child clauses, never the userId clause.

## Route delta (`src/lib/server/http/routes/events.ts`)

`feedQuerySchema` no longer declares source / kind / game (zod sees only the first repeated value, so it would silently drop B/C in `?source=A&source=B&source=C`). The handler reads multi-value axes via `c.req.queries('source' | 'kind' | 'game')`, validates each kind value against `VALID_EVENT_KINDS` (defense-in-depth for malformed multi-value URLs), and passes the arrays through to `listFeedPage`. Single-param callers receive a one-element array which the service helper collapses to `eq()` — zero query-plan regression.

The schema also gains an `all: "1" | "0"` field used by the SSR loader's default-30d toggle (the API itself ignores it; the API treats absent `from` / `to` as no filter).

## /feed page-server delta

`url.searchParams.getAll(...)` replaces `.get(...)` for source / kind / game. When no `from` / `to` / `all=1` is present, the loader synthesises `from = today - 30d, to = today` so the default-30d window flows through `listFeedPage`. The return shape adds:

- `activeFilters.source: string[]` (always present, possibly empty)
- `activeFilters.kind: string[]`
- `activeFilters.game: string[]`
- `activeFilters.defaultDateRange: boolean` (true only when no from/to/all params; lets the UI render the chip as `"Last 30 days (default)"` rather than the user-applied range copy)
- `activeFilters.all: boolean` (true when `?all=1`)

Plan 02.1-14's `deletedEvents` and the layout-passed `retentionDays` are preserved in the loader return shape.

## New component: `<DateRangeControl>`

Five preset chips (Today, Last 7 days, Last 30 days, All time, Custom…) + a `<details>`-style custom expander with two date inputs and an Apply button. The currently-active preset is matched by comparing `activeFilters.from / to` against today / today-Nd; matching presets get `aria-pressed="true"` and accent border. The "All time" preset emits `{ all: true }`, which the page-svelte handler turns into `?all=1` (opt-out of the 30-day default).

## Rewritten components

**`<FilterChips>`:** the `chips` derivation now iterates over `filters.source / kind / game` arrays and emits one chip per VALUE. The dismiss × passes `chip.value` so the parent's URL rebuild removes only that value (axis stays if other values remain). Date-range becomes a single chip with `axis === "dateRange"`; dismiss → opt-out via `?all=1`.

**`<FiltersSheet>`:** three `<select>` blocks become three `<fieldset>` checkbox lists. Source + game lists carry a typeahead `<input type="search">` to keep long lists scannable. Local state uses `Set<string>` reassigned on each toggle (immutable update so derived `filteredSources` recomputes). The from/to date inputs are removed — date is owned by `<DateRangeControl>` now. `onApply` signature changed to pass `string[]` for the multi-value axes.

**`/feed/+page.svelte`:** new `applyDateRange` / `dismissAxis(axis, value?)` / `applyFiltersFromSheet` handlers. The HARD CONTRACT from Plan 02.1-16 was honored: Plan 02.1-14's `<DeletedEventsPanel>` block + import are preserved exactly where Plan 14 placed them. `clearAll` now navigates to `/feed?all=1` so the 30-day default doesn't immediately re-apply.
