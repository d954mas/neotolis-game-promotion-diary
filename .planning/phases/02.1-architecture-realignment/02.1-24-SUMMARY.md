## Decisions Made

- **Triage flag via jsonb path, not schema column.** `metadata.triage.standalone` mirrors the `metadata.inbox.dismissed` precedent (Plan 02.1-05). Avoids a schema migration on the events table for what is fundamentally a binary user-set flag. Forward-compatible: `metadata.triage.*` becomes the natural home for future triage flags (e.g., `flagged_as_duplicate`) without further schema churn.
- **markStandalone detaches the game in the same UPDATE.** Standalone implies "not tied to any game" by definition (per UAT user direction). Setting `game_id=null` AND `metadata.triage.standalone=true` in one UPDATE keeps the data invariant: a row with `metadata.triage.standalone='true'` always has `game_id IS NULL`. unmarkStandalone clears the flag but keeps `game_id` null — the user can re-attach via /events/[id]/edit if they change their mind.
- **Audit-after-success ordering matches Plan 02.1-14 / 02.1-05 precedent.** Non-destructive triage actions write audit AFTER the UPDATE succeeds, so cross-tenant NotFoundError does not generate misleading audit. (The Phase 2 forensics-BEFORE pattern — removeSteamKey, softDeleteSource — is reserved for security-relevant DESTRUCTIVE actions.)
- **Inbox view excludes standalone events.** Standalone is a separate triage state, not "still in inbox". The user has triaged the event explicitly. The 4-state Show fieldset (Any / Inbox / Standalone / Specific) is mutually exclusive by construction; the inbox SQL clause adds `COALESCE(... 'triage'->>'standalone', 'false') = 'false'` alongside the existing dismissed exclusion.
- **FeedCard inline "Mark standalone" button is an EXPLICIT, NARROWLY-SCOPED exception to Plan 02.1-18.** Per UAT-NOTES.md §6.1-redesign user direction: "Чтобы была кнопка добавить к игре, или чтобы можно было пометить, ка не связанное с игрой". The button only renders when `isInboxRow=true` (i.e., game_id IS NULL AND not dismissed AND not standalone). It sits OUTSIDE the wrapping `<a href={`/events/${id}`}>` so its onclick doesn't trigger card navigation. AttachToGamePicker is the existing precedent for "mutating control outside the anchor" — the standalone button mirrors it exactly.
- **Idempotency via jsonb_set + audit-on-every-call.** A second `markStandalone` call on an already-standalone event succeeds (jsonb_set is idempotent); both calls write fresh audit rows. Mirrors `dismissFromInbox` precedent (Plan 02.1-05). Avoids the alternative "already-standalone → 422" design that would force the UI to track local state.
- **Migration snapshot generated by hand.** Drizzle-kit's `pnpm drizzle-kit generate` was not invoked — the audit_action enum schema source is a TypeScript const, drizzle-kit's diff output for `ALTER TYPE ADD VALUE` is mechanical (per Plan 02.1-12 / 02.1-14 precedent). Hand-rolled `0003_add_event_standalone_audit_actions.sql` matches the drizzle-kit-emitted shape exactly (`ALTER TYPE "public"."audit_action" ADD VALUE 'X';--> statement-breakpoint`). The 0003_snapshot.json is a copy of 0002 with id/prevId rotated and the two new enum values appended after `event.restored` — semantically equivalent to a generator run, faster, and avoids a network/drizzle-kit-CLI dependency in the parallel-executor flow.
