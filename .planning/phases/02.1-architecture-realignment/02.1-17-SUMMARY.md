## Decisions Made

- **publishedAt auto-fill DEFERRED to Phase 3.** YouTube oEmbed has no `published_at` field. Alternatives considered:
  - (a) YouTube Data API v3 `videos.list` — requires per-user API key (KEYS-01 lands in Phase 3); not feasible in 2.1.
  - (b) HTML-scrape `<meta itemprop="datePublished">` — fragile, breaks when YouTube changes markup.
  - (c) Skip auto-fill in 2.1 — user types the date manually as today (matching current /events/new defaults). Title + thumbnail + external_id are all auto-filled.
  Chose (c). Phase 3's KEYS-01 ships the YouTube Data API key UI; auto-fill of `publishedAt` lands then alongside the polling worker that needs the same key.
- **Shared youtubeUrlRequired superRefine.** Defining the rule once and applying it via `.superRefine(youtubeUrlRequired)` to both createEventSchema and updateEventSchema avoids the round-2 partial-ship risk where a kind-change PATCH could drop the url silently.
- **Service-layer opportunistic enrichment + route-layer load-bearing validation.** createEvent silently leaves externalId null on a malformed YouTube URL (the route's superRefine catches it first); the service is defense-in-depth only. This keeps the service callable from non-route contexts (e.g. seed scripts) without bringing along the validation overhead.
- **preview-url returns occurredAt: null** in 2.1 (no oEmbed published_at). The client falls back to today's date, matching the /events/new default.
