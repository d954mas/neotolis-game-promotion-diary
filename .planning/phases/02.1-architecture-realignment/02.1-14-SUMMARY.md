# Phase 02.1 Plan 14: Soft-Delete Event Recovery Summary

Closes VERIFICATION.md Gap 2 (P0 — soft-delete event recovery UI is missing). Delivers on the user-facing 60-day retention promise that `confirm_event_delete` has been making since Plan 02.1-09. Three layers ship as one cohesive feature: service-layer `restoreEvent` + `listDeletedEvents` (tenant-scoped, retention-window-bounded), HTTP routes (`PATCH /api/events/:id/restore` + `GET /api/events/deleted` with cross-tenant 404 + anonymous-401 sweep coverage), and `<DeletedEventsPanel>` below the CursorPager on `/feed` (KindIcon + strikethrough title + RetentionBadge + Restore button per row).

## What This Plan Built

The Phase 2.1 schema's `events.deletedAt` column was wired by Plan 02.1-01 but never surfaced in API or UI. Plan 02.1-09 shipped the soft-delete confirm dialog with copy that promised "you can restore it within 60 days" — until this plan, that copy was a lie. Plan 02.1-14 makes the promise truthful.

The service layer is the load-bearing tier:

- `restoreEvent(userId, eventId, ipAddress, userAgent?)` — clears `deletedAt` only when the row matches `(userId AND id AND deleted_at IS NOT NULL AND deleted_at >= now() - RETENTION_DAYS days)`. Cross-tenant / never-deleted / past-retention all collapse to `NotFoundError` by construction (the UPDATE returns no rows; the post-UPDATE `if (!row) throw new NotFoundError()` fires).
- `listDeletedEvents(userId)` — returns rows the user can still restore, sorted by `deletedAt DESC`. The retention-window cutoff filter excludes past-retention rows (those are pending Phase 6 purge; the UI never surfaces them).

The HTTP route `PATCH /api/events/:id/restore` is a thin wrapper over the service: extracts `c.var.userId` (set by `tenantScope` middleware), forwards to `restoreEvent`, projects the result through `toEventDto`. Cross-tenant 404 by construction (CLAUDE.md invariant 2 — body never contains "forbidden" or "permission"). The companion list route `GET /api/events/deleted` delegates to `listDeletedEvents` with the same DTO discipline.

The UI panel is collapsed by default (mirroring the `/sources` soft-deleted-section pattern from Plan 02.1-08) and surfaces below the CursorPager regardless of whether the main feed is empty. The Restore button POSTs the PATCH and calls `invalidateAll()` on 200 — the row disappears from the panel and reappears in the main feed without a page reload.

## Service Functions

```typescript
// src/lib/server/services/events.ts
export async function listDeletedEvents(userId: string): Promise<EventRow[]>;

export async function restoreEvent(
  userId: string,
  eventId: string,
  ipAddress: string,
  userAgent?: string,
): Promise<EventRow>;
```

Both `userId`-first per CLAUDE.md invariant 1. Both queries `eq(events.userId, userId)` is the first AND clause. `restoreEvent` writes `event.restored` audit row AFTER the UPDATE succeeds (rationale: non-destructive action; cross-tenant attempts must not generate misleading audit rows).

## Privacy Invariants Re-asserted (CLAUDE.md)

- **Invariant 1 (tenant scoping):** `restoreEvent` and `listDeletedEvents` are `userId`-first; `eq(events.userId, userId)` is the first AND clause in both. The custom ESLint rule `tenant-scope/no-unfiltered-tenant-query` cannot fire.
- **Invariant 2 (cross-tenant 404 not 403):** `restoreEvent` throws `NotFoundError` for cross-tenant / never-deleted / past-retention; `mapErr` translates to `{error: "not_found"}` status 404. Body never contains "forbidden" or "permission".
- **Invariant 3 (anonymous-401 sweep):** both new routes added to `MUST_BE_PROTECTED` allowlist; explicit per-route 401 tests in addition to the sweep.
- **Invariant 4 (audit INSERT-only):** `writeAudit` is the only write path; `event.restored` is the new audit verb.
- **Invariant 5 (DTO discipline):** routes project through `toEventDto` (which strips `userId`); the panel reads only `id`, `kind`, `title`, `deletedAt` (all non-secret).
- **Invariant 6 (Pino redact paths):** unchanged. No new secret-shaped field name introduced.
- **Invariant 7 (no public dashboards):** `/feed` is auth-gated; the panel renders only inside the protected route.
- **Invariant 8 (self-host parity):** no `APP_MODE` branching introduced. Same behavior in SaaS + self-host.

## Deviations from Plan

None — plan executed as written. Three minor process notes:

1. **drizzle-kit auto-generated SQL with `ADD VALUE … BEFORE 'source.added'`** rather than the plan's plain `ADD VALUE`. Both forms are functionally equivalent and the plan instructed to "overwrite if drizzle-kit emits differently". Final SQL hand-written to match the plan exactly.
2. **`drizzle-kit generate` initially appended a `0003_sparkling_whirlwind.sql` migration** because it noticed the schema diff (the AUDIT_ACTIONS const change had already been applied to the schema file before generation). Cleaned up by renaming the auto-generated snapshot to `0002_snapshot.json` (chains correctly with `0001_snapshot.json`'s id), removing the spurious `0003_*.sql`, and resetting the journal to three entries (idx 0/1/2).
3. **Local Node version mismatch** — local dev uses Node 20.15.0, but the project requires Node 22.11+. Tests fail to start without a Node 22+ binary on PATH. Worked around by using a cached Node 24 binary from npx and copying the rolldown native binding from the parent repo's `node_modules`. Self-check: CI uses Node 22 and the official path; this is purely a local-developer-machine issue.
